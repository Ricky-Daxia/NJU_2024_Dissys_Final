# 模块三：时间与同步

> 时间紧时只抓五件事：物理时钟为什么不可靠、Lamport 时钟怎么算、向量时钟如何判断因果/并发、分布式互斥、分布式选举。最新 PDF 把“同步与资源管理”里的互斥访问和选举机制列入范围。

## 1. 本模块主线

分布式系统没有可靠的全局时钟。不同机器的物理时钟会漂移，消息传输也有不确定延迟，所以不能简单用机器时间判断事件先后。

核心问题：

```text
分布式系统真正关心的通常不是墙上时间，而是事件之间有没有因果关系。
Lamport 时钟能保证因果关系对应时间戳递增，但不能反推出因果关系。
向量时钟可以判断两个事件是因果有序还是并发。
```

后端类比：

- 跨机器日志时间戳可能乱序。
- 链路追踪中 trace/span 关系通常比机器时间更可信。
- 多活复制、版本冲突检测可借鉴向量时钟思想。

## 2. 为什么物理时钟不可靠

标准答案：

```text
分布式系统中各节点使用各自的物理时钟，不同机器的时钟会因为晶振误差产生漂移。
同时，网络通信存在不可预测延迟，消息到达时间不能准确反映事件发生时间。
因此，仅依靠物理时钟无法可靠判断分布式事件的先后和因果关系。
逻辑时钟不追求真实时间，而是用于刻画事件的相对顺序和因果关系。
```

易错点：

- NTP 只能减小时钟偏差，不能彻底解决分布式因果顺序问题。
- 时间戳更大不一定表示真实发生得更晚。

## 3. Happens-before 关系

记号：

> `a -> b` 表示事件 a happens-before 事件 b，即 a 在因果上先于 b。

三条规则：

1. 同一进程内，若 a 发生在 b 前，则 `a -> b`。
2. 若 a 是发送消息事件，b 是接收该消息事件，则 `a -> b`。
3. 传递性：若 `a -> b` 且 `b -> c`，则 `a -> c`。

并发：

```text
如果既不能推出 a -> b，也不能推出 b -> a，则 a 和 b 是并发事件。
并发不一定表示物理上同时发生，只表示二者之间没有可证明的因果顺序。
```

## 4. Lamport 逻辑时钟

### 4.1 规则

必须会默写：

1. 本地事件：本进程时钟 `C = C + 1`。
2. 发送消息：先 `C = C + 1`，消息携带时间戳 `C`。
3. 接收消息：收到时间戳 `t` 后，设置 `C = max(C, t) + 1`。

### 4.2 性质

标准答案：

```text
Lamport 时钟用一个递增整数表示每个进程观察到的逻辑时间。
它满足：如果事件 a happens-before 事件 b，则 C(a) < C(b)。
但反过来不成立，C(a) < C(b) 不能说明 a 一定 happens-before b，因为两个并发事件也可能有不同时间戳。
Lamport 时钟可结合进程 ID 构造全序，但这个全序只是人为排序，不等于真实因果全序。
```

### 4.3 计算题套路

步骤：

1. 每个进程初始时钟通常从 0 开始。
2. 沿时间线从左到右处理事件。
3. 本地/发送事件直接加 1。
4. 接收事件用 `max(本地当前值, 消息时间戳) + 1`。
5. 最后检查每条消息是否满足发送时间戳小于接收时间戳。

小例子：

```text
P1 发送 m 时本地时钟从 2 加到 3，消息带 3。
P2 收到 m 前本地时钟为 5，则接收事件时间戳为 max(5, 3) + 1 = 6。
```

易错点：

- 发送消息也要先加 1。
- 接收消息不是简单加 1，而是 `max(local, message) + 1`。
- Lamport 时间戳相等或大小关系不能直接判断并发，判断并发要看 happens-before 或用向量时钟。

## 5. 向量时钟

### 5.1 规则

假设有 n 个进程，每个进程维护长度为 n 的向量。

必须会默写：

1. 本地事件：本进程分量加 1。
2. 发送消息：本进程分量加 1，并携带整个向量。
3. 接收消息：先逐位取 `max(本地向量, 消息向量)`，再将本进程分量加 1。

### 5.2 如何判断因果和并发

给两个向量时间戳 `V(a)` 和 `V(b)`：

```text
若 V(a) 每一维都 <= V(b)，且至少一维 <，则 a -> b。
若 V(b) 每一维都 <= V(a)，且至少一维 <，则 b -> a。
否则二者并发。
```

例子：

| V(a) | V(b) | 关系 |
| --- | --- | --- |
| [1, 2, 0] | [1, 3, 0] | a -> b |
| [2, 0, 1] | [1, 3, 1] | 并发 |
| [4, 2, 1] | [4, 2, 1] | 同一逻辑版本或无法区分先后 |

标准答案：

```text
向量时钟为每个进程维护一个分量，记录该进程已观察到的各进程事件数量。
与 Lamport 时钟相比，向量时钟不仅能保证因果关系对应时间戳递增，还能判断两个事件是否并发。
判断方法是逐维比较：若一个向量逐维小于等于另一个且至少一维严格小，则前者因果先于后者；若两者无法逐维包含，则事件并发。
```

易错点：

- 向量时钟的核心价值是判断因果和并发，不是构造全序。
- 接收消息时要先逐位取 max，再给本进程分量加 1。
- `[2,0]` 和 `[1,3]` 是并发，不是谁总和大谁更晚。

## 6. Lamport vs 向量时钟

| 项目 | Lamport 时钟 | 向量时钟 |
| --- | --- | --- |
| 数据结构 | 一个整数 | 一个向量 |
| 开销 | 小 | 随进程数增长 |
| 能保证 | `a -> b` 则 `C(a) < C(b)` | 可判断因果和并发 |
| 不能做什么 | 不能由时间戳大小反推因果 | 不适合进程数量极大且频繁变化的系统 |

标准对比答案：

```text
Lamport 时钟开销小，只用一个整数即可维护逻辑时间。
它能保证因果先后对应时间戳递增，但不能根据时间戳大小反推出因果关系。
向量时钟为每个进程维护一个分量，开销更大，但可以通过逐维比较判断两个事件是否存在因果关系或是否并发。
```

## 7. 物理时钟同步算法

### 7.1 Cristian 算法

场景：

> 客户端向时间服务器请求时间，用往返延迟估计当前时间。

步骤：

1. 客户端在 `T0` 发送请求。
2. 服务器收到后返回当前时间 `Ts`。
3. 客户端在 `T1` 收到响应。
4. 若假设网络延迟近似对称，则客户端可设置时间为 `Ts + (T1 - T0) / 2`。

标准答案：

```text
Cristian 算法通过向时间服务器请求当前时间来同步物理时钟。
客户端记录请求发送和响应接收时间，估计往返时间 RTT，并假设单向延迟约为 RTT/2。
收到服务器时间 Ts 后，客户端将本地时间调整为 Ts + RTT/2。
该算法简单，但精度依赖网络延迟估计，延迟不对称时误差会增大。
```

### 7.2 平均时间同步

标准答案：

```text
平均时间同步中，各节点相互交换本地时间，收集其他节点的时钟值。
节点可丢弃明显异常值，再对剩余时间取平均或加权平均，并将本地时钟向该平均值调整。
这种方法不依赖单一时间服务器，但需要处理消息延迟和异常节点影响。
```

题目若要求“同步后时间为所有节点平均值”，可按下面流程写：

```text
选择一个 master 作为协调者。master 向所有 slave 发送时间请求，并记录请求发送和响应接收时间，用于估计通信延迟。
各 slave 返回本地时钟值。master 收集所有节点时间，必要时剔除明显异常值，并把自身时间也纳入计算。
master 计算平均时间或每个节点相对平均时间的偏移量。
最后 master 通知每个节点应调整的偏移量，各节点通过逐步调快或调慢本地时钟靠近平均时间。
```

易错点：

- master 自己也要纳入平均并调整。
- 更稳妥的说法是发送“偏移量”而不是直接把所有机器硬设置为同一个时间，避免时间倒退造成异常。

### 7.3 Berkeley 算法

```text
Berkeley 算法通常由一个时间守护进程作为协调者。
协调者轮询各节点的本地时间，估计并修正消息延迟后计算平均时间或平均偏差，再通知各节点把时钟向该平均值调整。
它不依赖外部 UTC 时间源，更适合让一个封闭系统内部时钟彼此一致。
```

运行步骤：

1. 时间守护进程定期向各机器询问当前时间。
2. 各机器返回自己的本地时钟值。
3. 守护进程估计消息延迟，并剔除明显异常的时钟值。
4. 守护进程计算平均时间或相对平均值的偏差。
5. 守护进程向每台机器发送应调整的偏移量，而不是简单广播一个绝对时间。

答题模板：

```text
Berkeley 算法用于让一个分布式系统内部各节点时钟保持相对一致。
系统中有一个时间守护进程作为协调者，它周期性轮询各节点的本地时间，估计通信延迟并剔除异常值后计算平均时间。
随后协调者把每个节点应该调整的偏移量发送给对应节点，使所有节点逐渐向平均时间靠拢。
它与 Cristian 算法不同，Berkeley 不要求存在外部准确 UTC 时间服务器，而是追求系统内部一致。
```

易错点：

- Berkeley 算法不是让所有机器直接设置成协调者的时间，而是根据平均偏差调整。
- 它解决的是物理时钟近似同步，不解决 happens-before 因果判断。

### 7.4 NTP

```text
NTP 使用分层时间服务器同步物理时钟。
低层服务器从高层服务器获取时间并继续向下提供时间服务，客户端通过多次采样估计网络延迟和时钟偏移。
NTP 能减小互联网规模系统中的时钟偏差，但仍不能用来可靠判断分布式事件的因果顺序。
```

知识点展开：

```text
NTP 使用层级结构组织时间服务器。靠近真实时间源的服务器位于较高层，低层服务器从高层同步时间，
普通主机再从一个或多个时间服务器同步。客户端通过多次交换时间戳估计网络延迟和本地时钟偏移，
并逐步调整本地时钟，避免时间突然跳变带来异常。
```

对比题模板：

```text
Cristian 算法依赖一个时间服务器返回当前时间，客户端用 RTT/2 估计单向延迟；
Berkeley 算法由协调者收集多个节点时间并计算平均偏差，强调系统内部时钟一致；
NTP 则面向大规模网络，使用分层时间服务器和多次采样来同步互联网主机时间。
三者都只能减小物理时钟误差，不能取代 Lamport 时钟或向量时钟来判断因果顺序。
```

### 7.5 因果有序组播

```text
因果有序组播要求：如果消息 m1 happens-before 消息 m2，那么任何进程都必须先交付 m1，再交付 m2。
如果两个消息并发，则不同进程可以用不同顺序交付。
实现上通常借助向量时钟或依赖关系记录，接收方在缺少前置因果消息时暂缓交付后续消息。
```

知识点展开：

```text
组播中“收到消息”和“交付给应用”不是一回事。
节点可以先收到某条消息，但如果它依赖的因果前置消息还没到，就不能立刻交付给上层应用，而要暂存在缓冲区。
等前置消息到达并交付后，才能交付这条后续消息。
这样可以保证所有进程观察到的因果相关消息顺序一致。
```

小例子：

```text
P1 先组播 m1：“变量 x 已更新”。
P2 收到 m1 后，又组播 m2：“我基于新 x 计算了 y”。
那么 m1 -> m2。任何进程即使先收到 m2，也必须等 m1 交付后才能交付 m2。
如果 m1 和 m2 分别由两个互不通信的进程同时发出，它们并发，则因果组播允许不同进程以不同顺序交付。
```

答题模板：

```text
因果有序组播要求所有进程按照 happens-before 关系交付组播消息。
如果 m1 因果先于 m2，则任何正确进程都不能先交付 m2 再交付 m1；
如果两条消息并发，则协议不强制统一顺序。
实现时可为消息附带向量时钟或依赖信息，接收方发现缺少前置消息时先缓冲，等依赖满足后再交付。
```

易错点：

- 因果有序组播不要求所有消息全序；并发消息可以不同序。
- “接收顺序”不等于“交付顺序”，协议控制的是交付给应用的顺序。

## 8. 分布式互斥访问

问题：

> 多个进程分布在不同机器上，没有共享内存和统一内核锁，仍需要互斥访问共享资源。

互斥算法要满足：

- Safety：任意时刻最多一个进程进入临界区。
- Liveness：请求进入临界区的进程最终能进入，不能永久饿死。
- Ordering/Fairness：最好按请求顺序或逻辑时间顺序进入。

### 8.1 集中式互斥

```text
集中式互斥使用一个 coordinator 管理临界区。
进程进入临界区前向 coordinator 请求许可；coordinator 若空闲则授权，否则排队。
进程退出临界区后通知 coordinator，coordinator 再授权队列中的下一个进程。
优点是简单、消息少；缺点是 coordinator 是单点故障和性能瓶颈。
```

### 8.2 分布式互斥

```text
分布式互斥不依赖单一协调者，进程需要和其他进程交换请求与应答。
典型思路是用 Lamport 时间戳给请求排序，进程只有在自己的请求排在前面并收到其他进程许可后才能进入临界区。
优点是没有中心瓶颈；缺点是消息开销大，并且节点故障会影响协议进展。
```

### 8.3 令牌环互斥

```text
令牌环互斥把进程组织成逻辑环，系统中只有一个 token。
只有持有 token 的进程才能进入临界区；使用完后把 token 传给下一个进程。
它天然保证互斥，但 token 丢失或持有 token 的节点故障需要额外恢复机制。
```

三类互斥算法对比：

| 算法 | 进入临界区消息数 | 优点 | 主要问题 |
| --- | --- | --- | --- |
| 集中式 | 通常 3 条：请求、授权、释放 | 简单、消息少 | coordinator 单点故障 |
| 分布式 | 通常约 `2(N-1)` 条：请求和应答 | 无中心瓶颈 | 消息多，任一节点故障会拖住进展 |
| 令牌环 | 等待 token 到达 | 天然互斥、无请求广播 | token 丢失和节点故障要恢复 |

答题展开：

```text
集中式互斥把进入临界区的决定权交给 coordinator。请求者先发 request，coordinator 空闲时发 grant，
进程退出临界区后发 release。它消息数少、实现简单，但 coordinator 故障会导致系统无法授权。
分布式互斥没有中心节点，进程向其他所有进程广播带 Lamport 时间戳的请求，并等待所有进程回复 OK。
只有当自己的请求在全局排序中最靠前且收到全部许可时才能进入临界区。它避免了中心瓶颈，但消息多且对节点故障敏感。
令牌环互斥让唯一 token 在逻辑环上传递，只有持有 token 的进程可进入临界区。它自然保证安全性，但 token 丢失或持有者故障时需要检测和重建。
```

考试比较模板：

```text
三种互斥算法都要保证 safety 和 liveness。
集中式算法消息少、实现简单，但有单点故障；
分布式算法无中心瓶颈，但每次进入临界区要与其他节点交换消息，开销大且节点故障会影响进展；
令牌环算法用唯一 token 保证互斥，正常情况下不需要广播请求，但要额外处理 token 丢失和环上节点故障。
```

考题抓手：

```text
回答互斥题时先写目标：保证多个分布式进程不会同时进入临界区。
再按算法说明请求、授权、进入、释放四步。
最后写缺点：集中式有单点故障，分布式消息开销大，令牌环要处理 token 丢失。
```

## 9. 分布式选举机制

问题：

> 多个节点需要选出一个 coordinator / leader，用于协调资源、提交事务或管理成员。

### 9.1 Bully 算法

```text
Bully 算法假设每个进程有唯一编号，编号越大优先级越高。
当进程发现 coordinator 失效时，向所有编号更高的进程发送 election 消息。
如果没有更高编号进程响应，它宣布自己成为 coordinator；
如果有更高编号进程响应，则由更高编号进程继续发起选举。
最终存活的最高编号进程成为 coordinator。
```

### 9.2 Ring 算法

```text
Ring 选举算法把进程组织成逻辑环。
检测到 leader 失效的进程沿环发送 election 消息，消息中记录候选进程编号。
消息绕环一周后，最高编号的存活进程被选为 coordinator，并通过 coordinator 消息通知所有节点。
```

易错点：

- 选举算法解决“谁当协调者”，不是直接解决数据一致性。
- Bully 依赖进程编号和能联系更高编号节点；Ring 依赖逻辑环传递消息。

## 10. 考前最小背诵清单

- 物理时钟不可靠：时钟漂移 + 网络延迟不确定。
- happens-before 三规则：同进程顺序、发送先于接收、传递性。
- Lamport 规则：本地 +1，发送 +1 带时间戳，接收 `max(local,msg)+1`。
- Lamport 性质：`a -> b` 推出 `C(a) < C(b)`，反过来不成立。
- 向量规则：本地/发送本分量 +1，接收逐位 max 后本分量 +1。
- 向量判断：逐维小于等于且至少一维小，表示因果先后；互不包含表示并发。
- 物理时钟同步：Cristian 用服务器时间 + RTT/2；Berkeley 算平均偏差；NTP 是分层同步。
- 因果有序组播：有因果关系的消息必须按因果顺序交付，并发消息可乱序。
- 分布式互斥：集中式、分布式、令牌环；目标是 safety 和 liveness。
