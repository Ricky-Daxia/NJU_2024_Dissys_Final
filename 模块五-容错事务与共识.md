# 模块五：容错、恢复与分布式提交

> 时间紧时优先抓最新 PDF 明确列出的内容：可信系统特征、提高可信性的途径、K 容错与拜占庭问题、系统恢复、checkpoint、2PC/3PC。Paxos/Raft 保留为往年题补充。

## 1. 本模块主线

分布式系统一定会遇到故障。最新 PDF 把本模块分成两条主线：容错/恢复，以及分布式提交。

- 容错：节点或消息出错时，系统如何继续提供正确服务。
- 恢复：故障发生后，系统如何回到正确状态。
- 事务提交：多个参与者如何对提交/中止达成一致结果。
- 共识：往年题补充，多个节点如何对某个值或日志顺序达成一致。

核心句：

```text
可信系统关注 availability、reliability、safety、maintainability。
恢复可分为回退恢复和前向恢复，checkpoint 用于减少回退范围。
2PC 解决原子提交，但协调者故障可能导致阻塞。
3PC 试图通过增加阶段减少阻塞，但假设更强、实现更复杂。
Paxos/Raft 是往年题补充，核心依赖多数派交集避免脑裂。
多数派协议保证一致性，但网络分区时少数派通常不可用。
```

## 2. 可信系统 Dependable System

四个常见特征：

| 特征 | 含义 |
| --- | --- |
| Availability | 系统在某一时刻可用的概率 |
| Reliability | 系统在一段时间内持续正确工作的概率 |
| Safety | 系统故障时不造成灾难性后果 |
| Maintainability | 系统故障后易于修复和恢复 |

标准答案：

```text
可信系统是指在故障存在时仍能让用户对其服务有信心的系统。
常见特征包括可用性、可靠性、安全性和可维护性。
可用性关注系统某时刻是否可提供服务；可靠性关注一段时间内是否持续正确服务；
安全性关注故障是否会造成严重后果；可维护性关注故障后定位、修复和恢复的难易程度。
```

提高可信性的途径：

```text
提高系统可信性通常包括故障预防、故障容忍、故障移除和故障预测。
故障预防通过良好设计避免故障引入；故障容忍通过冗余和恢复机制在故障发生时继续服务；
故障移除通过测试、调试和修复减少故障；故障预测通过监控和分析提前评估故障风险。
```

### 2.1 故障、错误和失效

```text
Fault 是故障根因，例如硬件损坏、软件 bug 或网络问题。
Error 是系统内部出现的错误状态。
Failure 是系统对外提供的服务偏离了正确规范。
三者关系可以理解为：fault 可能导致 error，error 未被处理时可能表现为 failure。
```

答题模板：

```text
故障、错误和失效是三个层次。
fault 是导致问题的根因，例如硬件损坏、程序缺陷或网络异常；
error 是 fault 激活后在系统内部形成的错误状态；
failure 是系统对外服务偏离规范，被用户或其他组件观察到。
容错机制的目标是在 error 变成 failure 之前检测、屏蔽或恢复它。
```

### 2.2 故障模型

| 模型 | 含义 |
| --- | --- |
| Crash failure | 节点停止工作，不再发送任何消息 |
| Omission failure | 应发送或应接收的消息被遗漏 |
| Timing failure | 响应超过规定时间，实时约束被破坏 |
| Response failure | 响应内容错误或状态转换错误 |
| Arbitrary/Byzantine failure | 节点可任意错误，包括说谎、发送矛盾消息 |

答题时先说明故障模型，再套副本数或协议性质。

故障模型展开：

| 模型 | 例子 | 协议影响 |
| --- | --- | --- |
| Crash | 服务器直接宕机、不再响应 | 可通过超时、主备切换处理 |
| Omission | 请求或响应被丢弃 | 需要重传、确认、可靠通信 |
| Timing | 响应太慢，超过实时系统 deadline | 需要时间假设、超时策略 |
| Response | 返回错误值或执行错误状态转换 | 需要校验、投票或恢复 |
| Byzantine | 对不同节点说不同话，甚至伪造行为 | 需要拜占庭容错，副本数要求更高 |

标准答案：

```text
故障模型描述系统中组件可能以什么方式出错。
Crash failure 表示节点停止工作；omission failure 表示消息发送或接收被遗漏；
timing failure 表示响应时间违反约束；response failure 表示响应内容或状态转换错误；
arbitrary/Byzantine failure 最强，节点可以任意错误，甚至向不同节点发送矛盾信息。
不同故障模型需要不同容错机制，不能在未说明故障模型时直接套副本数公式。
```

### 2.3 冗余方式

| 冗余类型 | 含义 | 例子 |
| --- | --- | --- |
| Information redundancy | 增加额外信息用于检测或纠错 | 校验码、纠删码 |
| Time redundancy | 失败后重试或重新执行 | RPC 重试、任务重跑 |
| Physical redundancy | 增加硬件、进程或副本 | 多副本、主备、集群 |

答题模板：

```text
提高可信性常依赖冗余。
信息冗余是在数据中加入额外信息，用于检测或纠正错误，例如校验码、奇偶校验和纠删码；
时间冗余是通过重新执行或重试来掩盖临时故障，例如消息重传和任务重跑；
物理冗余是增加额外硬件、进程或副本，例如主备服务器、多副本存储和集群部署。
冗余能提高可靠性和可用性，但会增加成本，并可能引入一致性维护问题。
```

## 3. 故障模型与副本数量

### 3.0 `K+1`、`2K+1`、`3K+1` 的边界

| 目标/故障模型 | 容忍 K 个故障常见数量 | 直觉 |
| --- | --- | --- |
| Fail-silent / crash | `K+1` | K 个坏了，至少还剩 1 个正确服务 |
| 唯一正确结果 + 多数投票 | `2K+1` | K 个错误值压不过 K+1 个正确值 |
| Byzantine agreement | `3K+1` | 错误节点可能对不同正确节点说不同话，需要更多正确节点达成一致 |

PPT 中“fail-silent faults: K+1，Byzantine faults: 2K+1”对应的是进程组通过副本输出继续提供服务的直觉；往年题里的传感器平均值/拜占庭协同要求正确节点之间达成一致，按 `3K+1` 回答更稳。

答题时这样区分：

```text
第一步先说故障模型。
如果节点只是崩溃或停止响应，为了保证仍有一个正确副本可服务，K+1 个副本可容忍 K 个 crash/fail-silent 故障。
如果节点可能返回错误值，但所有正确节点返回同一个唯一正确结果，则用多数投票，至少需要 2K+1。
如果错误节点能对不同正确节点发送不同信息，并要求所有正确节点达成一致，则属于拜占庭一致问题，常见要求 3K+1。
```

例题判断：

| 题目描述 | 应用公式 |
| --- | --- |
| “最多 K 台机器宕机，至少还有服务可用” | `K+1` |
| “最多 K 个副本返回随机错误值，正确结果唯一，用多数投票” | `2K+1` |
| “故障节点可能向不同节点发送不同值，正确节点要一致决定” | `3K+1` |

### 3.1 `2K + 1`

适用场景：

> 正确节点的结果唯一且相同，错误节点最多 K 个，错误结果可通过多数投票压过。

标准答案：

```text
如果系统中最多有 K 个节点可能返回错误值，但所有正确节点都会返回同一个正确结果，
则至少需要 2K+1 个副本。
因为即使 K 个节点出错，仍有 K+1 个正确节点，多数投票可以选出正确结果。
```

例子：

- 最多 1 个错误节点，需要 3 个副本。
- 最多 2 个错误节点，需要 5 个副本。

### 3.2 `3K + 1`

适用场景：

> 拜占庭故障：错误节点可能说谎、发送不同信息、互相串通，正确节点需要达成一致。

标准答案：

```text
如果错误节点可能表现为拜占庭故障，即向不同节点发送不同信息，并要求所有正确节点对结果达成一致，
则常见要求至少 3K+1 个节点才能容忍 K 个故障节点。
这种情况下不仅要压过错误结果，还要保证正确节点之间能通过多轮交换识别一致信息并达成相同结论。
```

易错点：

```text
2K+1 和 3K+1 不能脱离故障模型背诵。
只需要多数投票选唯一正确结果时是 2K+1；
涉及拜占庭式说谎、不同节点看到不同消息并要求正确节点一致时，常见是 3K+1。
```

## 4. 系统恢复

### 4.1 回退恢复 Rollback Recovery

```text
回退恢复是在检测到故障后，将系统状态恢复到之前保存的正确状态，然后从该状态重新执行。
它通常依赖 checkpoint 和日志。
优点是思路简单、适合无法精确修复当前错误状态的场景；缺点是会丢失 checkpoint 之后的部分工作。
```

### 4.2 前向恢复 Forward Recovery

```text
前向恢复是在检测到错误后，不回到过去状态，而是通过补偿、修复或转换把系统推进到新的正确状态。
它适合某些可通过补偿操作修复的应用场景。
优点是可能减少重复执行；缺点是需要针对错误设计修复逻辑，实现更复杂。
```

### 4.3 Checkpoint

```text
Checkpoint 是系统在运行过程中保存的一致状态快照。
发生故障后，系统可以从最近 checkpoint 恢复，而不必从头开始。
Checkpoint 的作用是限制回退恢复的范围，但保存 checkpoint 本身有存储和性能开销。
```

PPT 补充：

| checkpoint 方式 | 含义 | 风险/代价 |
| --- | --- | --- |
| Independent checkpointing | 各进程独立保存本地 checkpoint | 可能形成不一致全局状态，产生 domino effect |
| Coordinated checkpointing | 多进程协调生成一致全局 checkpoint | 协调开销更大，但恢复简单 |
| Two-phase checkpointing | 先准备并冻结相关状态，再统一确认 checkpoint | 类似两阶段思想，可靠但可能阻塞 |

知识点展开：

```text
分布式 checkpoint 的难点是多个进程之间存在消息依赖。
如果各进程独立保存 checkpoint，恢复时可能出现某个进程已经记录“收到消息”，但发送方恢复到发送该消息之前的状态，
这种全局状态不一致，会导致级联回滚，也叫 domino effect。
协调式 checkpoint 让多个进程协商生成一个一致的全局快照，恢复时可以一起回到该快照。
两阶段 checkpoint 则先让进程进入准备状态并保存临时 checkpoint，再由协调者统一决定提交或放弃这次 checkpoint。
```

答题模板：

```text
独立 checkpoint 开销小、无需全局协调，但可能产生不一致全局状态和 domino effect；
协调 checkpoint 通过让相关进程共同生成一致快照，使恢复更简单，但运行时协调成本更高；
两阶段 checkpoint 借鉴两阶段思想，先准备临时 checkpoint，再统一确认，可提高一致性，但协调者故障时可能带来阻塞。
```

考题抓手：

```text
回退恢复：回到过去正确状态再执行。
前向恢复：从当前错误状态修到未来正确状态。
Checkpoint：保存恢复点，降低故障后的重做成本。
```

## 5. 2PC：两阶段提交

解决问题：

> 多个参与者要么全部提交，要么全部中止，保证分布式事务的原子提交。

角色：

- Coordinator：协调者。
- Participant：参与者。

### 5.1 正常流程

第一阶段：投票 / 准备阶段。

1. 协调者向所有参与者发送 prepare / vote-request。
2. 参与者检查自己能否提交。
3. 能提交则写入本地日志并回复 yes。
4. 不能提交则回复 no。

第二阶段：决定阶段。

1. 如果所有参与者都 yes，协调者写 commit 日志并广播 commit。
2. 如果任一参与者 no 或超时，协调者写 abort 日志并广播 abort。
3. 参与者收到决定后执行 commit/abort，并释放资源。

标准答案：

```text
2PC 分为投票阶段和决定阶段。
投票阶段中，协调者询问所有参与者是否可以提交，参与者若可以提交则写入准备日志并回复 yes，否则回复 no。
决定阶段中，如果所有参与者都回复 yes，协调者广播 commit；只要有参与者回复 no 或超时，协调者广播 abort。
参与者根据协调者决定提交或中止事务，从而保证所有参与者结果一致。
```

### 5.2 为什么会阻塞

关键场景：

- 参与者已经投 yes，并进入 prepared 状态。
- 此时协调者宕机。
- 参与者不知道全局决定是 commit 还是 abort。
- 为避免破坏原子性，只能等待协调者恢复或从其他节点得知决定。

标准答案：

```text
2PC 的主要问题是阻塞。
当参与者投 yes 后进入 prepared 状态，它已经承诺可以提交，不能单方面中止。
如果此时协调者故障，而参与者尚未收到 commit 或 abort 决定，就无法判断全局事务结果。
为了保证原子性，参与者只能阻塞等待协调者恢复或获取最终决定。
```

易错点：

- 2PC 不是共识协议，它依赖协调者做最终决定。
- 2PC 保证原子提交，但不保证非阻塞。
- 参与者投 yes 后不能随便 abort，否则可能和其他已 commit 节点不一致。

### 5.3 2PC 参与者故障恢复判断

PPT 里常考参与者恢复后根据日志状态判断能否单方面决定：

| 参与者恢复时状态 | 可以怎么做 | 原因 |
| --- | --- | --- |
| Initial | 可单方面 abort | 还没投 yes，没有承诺提交 |
| Ready/Prepared | 不能单方面决定，需询问协调者或其他参与者 | 已投 yes，可能全局已 commit |
| Commit | 执行 commit 并可告知其他节点 | 已知道全局决定 |
| Abort | 执行 abort 并可告知其他节点 | 已知道全局决定 |

```text
2PC 阻塞的核心状态是 ready/prepared。
参与者一旦投 yes，就承诺自己可以提交；如果此时没有收到最终 commit/abort，单方面 abort 可能破坏原子性。
```

恢复题答题模板：

```text
参与者恢复后应先查看本地日志。
如果日志中没有 ready/prepared 记录，说明它还没有投 yes，可以安全 abort。
如果日志中已有 commit 或 abort 记录，说明已经知道全局决定，按日志重做该决定即可。
最麻烦的是 ready/prepared 状态：参与者已经投 yes，但没有最终决定。
此时不能单方面 commit 或 abort，只能询问协调者或其他参与者；如果无法获知决定，就会阻塞。
```

易错点：

- prepared 状态不能因为超时就自行 abort；否则可能和已经收到 commit 的参与者不一致。
- 2PC 的阻塞不是正常流程复杂，而是特定故障状态下参与者缺少全局决定。

## 6. 3PC：三阶段提交

解决目标：

> 3PC 在 2PC 基础上增加阶段，试图减少参与者在协调者故障时的阻塞。

典型阶段：

1. CanCommit：询问参与者是否可能提交。
2. PreCommit：如果都可以提交，先进入预提交状态。
3. DoCommit：最终提交。

标准答案：

```text
三阶段提交把两阶段提交的决定过程拆得更细，通常包括 CanCommit、PreCommit 和 DoCommit。
第一阶段协调者询问参与者是否可以提交；第二阶段在所有参与者同意后发送 pre-commit，使参与者知道全局已准备提交；
第三阶段再发送 do-commit 完成提交。
相比 2PC，3PC 试图避免参与者在某些协调者故障场景下无限阻塞，但它依赖更强的超时和网络假设，实现也更复杂。
```

易错点：

- 3PC 不是绝对不会阻塞；它是在更强系统假设下减少阻塞风险。
- 考试如果只问 distributed commit，先写原子性目标，再写 2PC/3PC 的阶段和阻塞问题。

## 7. 往年题补充：Paxos

解决问题：

> 在可能发生节点故障和消息延迟的异步分布式系统中，让多个节点对一个值达成一致。

角色：

- Proposer：提出值。
- Acceptor：投票接受值。
- Learner：学习已选定的值。

核心思想：

```text
使用递增 proposal number 和多数派 quorum。
两个多数派必然相交，因此后续提案能通过交集节点知道之前可能已被选中的值，避免选出两个不同值。
```

### 7.1 基本流程

阶段一：Prepare / Promise。

1. Proposer 选择递增编号 n，向多数派 acceptor 发送 prepare(n)。
2. Acceptor 若 n 大于自己见过的编号，则承诺不再接受编号小于 n 的提案，并返回自己已接受过的最大编号提案。

阶段二：Accept / Accepted。

1. Proposer 收到多数派 promise 后，选择值。
2. 如果回复中已有被接受的值，必须选择编号最高的那个值；否则可选择自己的值。
3. Proposer 向多数派发送 accept(n, value)。
4. 多数派接受后，该值被选定。

标准答案：

```text
Paxos 通过 proposal number 和多数派机制保证共识安全性。
Proposer 先向多数派发送 prepare 请求，Acceptor 承诺不再接受更小编号的提案，并返回自己已接受过的最大编号提案。
Proposer 获得多数派 promise 后，如果发现已有被接受的值，就必须沿用编号最高的值；否则可以提出自己的值。
随后 Proposer 发送 accept 请求，当多数派接受同一提案后，该值被选定。
```

### 7.2 为什么可能活锁

标准答案：

```text
Paxos 可能出现活锁，因为多个 proposer 可同时提出更高编号的提案。
一个 proposer 刚获得部分 promise，另一个 proposer 又用更高编号抢占，使前一个提案无法继续完成。
如果多个 proposer 不断互相抢占，系统一直在处理更高编号提案，却迟迟不能选定值。
常见解决思路是选举一个稳定 leader，由 leader 统一发起提案，减少竞争。
```

易错点：

- 活锁不是死锁。系统一直在活动，只是没有进展。
- Paxos 的安全性是不会选出两个不同值；活性需要额外假设或 leader 稳定。
- 最新 PDF 未显式列 Paxos，优先级低于 2PC/3PC 和恢复，但往年题出现过，可作为补充分。

## 8. 往年题补充：Raft

Raft 三部分必须会：

1. Leader election：领导者选举。
2. Log replication：日志复制。
3. Safety：安全性。

### 8.1 基本角色

- Leader：处理客户端请求，复制日志。
- Follower：被动接收 leader/candidate 消息。
- Candidate：发起选举的候选者。

核心概念：

- Term：任期，逻辑时间。
- Log：日志，每条日志有 index 和 term。
- Majority：多数派。

### 8.2 Leader 选举

标准答案：

```text
Raft 中节点初始为 follower。
如果 follower 在选举超时时间内没有收到 leader 心跳，就转为 candidate，增加 term 并向其他节点请求投票。
节点在同一 term 内最多投一票，候选者获得多数票后成为 leader。
Leader 定期发送心跳维持权威；若发现更高 term，则节点转为 follower。
```

### 8.3 日志复制

标准答案：

```text
Raft 中客户端请求由 leader 处理。
Leader 将命令追加到本地日志，然后通过 AppendEntries RPC 复制给 follower。
当某条日志复制到多数派节点后，leader 将其提交并应用到状态机，再通知 follower 提交。
Follower 通过检查前一条日志的 index 和 term 与 leader 保持一致，若不一致则回退并修复日志。
```

### 8.4 安全性

标准答案：

```text
Raft 的安全性依赖任期、日志匹配和多数派交集。
一个 term 内最多只有一个 leader 能获得多数票；
已提交日志必须存储在多数派上，之后能当选的新 leader 也必须获得多数派支持，因此会与旧多数派相交；
选举限制要求候选者日志至少和投票者一样新，从而避免缺少已提交日志的节点成为 leader。
```

后端类比：

- etcd、Consul、ZooKeeper 类系统都依赖多数派避免脑裂。
- 少数派分区通常不能继续写入，因为没有多数派会破坏一致性。
- 最新 PDF 未显式列 Raft，但往年题出现过“Raft 三部分过程”，保留为补充题。

### 8.5 Raft 应用题解题模板

Raft 大题通常不会只考流程默写，而是给出一个集群场景，让你判断选举、日志复制、提交和故障恢复。答题时按下面顺序分析。

#### 8.5.1 固定分析顺序

1. 先看 `term`：任期更大的消息会让旧 leader/candidate 退回 follower。
2. 再看多数派：是否有超过半数节点参与投票或复制。
3. 再看日志新旧：候选者日志是否至少和投票者一样新。
4. 再看提交条件：leader 当前任期的日志复制到多数派后才能安全提交。
5. 最后看故障后行为：未提交日志可能被新 leader 覆盖；已提交日志必须被新 leader 保留。

一句话抓手：

```text
Raft 应用题就是围绕 term、majority、log up-to-date、commit index、conflict repair 五件事推。
```

#### 8.5.2 Leader 选举场景

常见题型：

```text
5 个节点 A/B/C/D/E，A 是 leader。
A 宕机后，B 和 C 同时超时发起选举，会发生什么？
```

答题模板：

```text
节点超时后会增加 term，转为 candidate，先给自己投票，再向其他节点发送 RequestVote。
每个节点在同一 term 内最多投一票。
如果某个 candidate 获得多数票，则成为 leader；如果多个 candidate 分票导致没人过半，则本轮选举失败。
随后节点等待随机 election timeout，再发起下一轮选举。随机超时用于降低持续分票概率。
```

易错点：

- 获得“最多票”不够，必须获得多数派。
- 同一 term 内一个节点最多投一票。
- 如果收到更高 term 的合法消息，candidate/leader 必须退回 follower。

#### 8.5.3 日志复制场景

常见题型：

```text
Leader 收到客户端写请求后，日志复制到哪些节点才算提交？
Follower 日志和 leader 不一致怎么办？
```

答题模板：

```text
客户端写请求先发给 leader。
leader 把请求追加为本地日志条目，再通过 AppendEntries 发送给 follower。
AppendEntries 中包含 prevLogIndex 和 prevLogTerm。
如果 follower 在对应位置找不到相同 term 的日志，就拒绝该请求；
leader 会回退 nextIndex，继续尝试更早的位置，直到找到共同前缀。
之后 follower 删除冲突日志，追加 leader 发来的日志。
当日志复制到多数派后，leader 推进 commitIndex，并把该日志应用到状态机，再通知 follower 提交。
```

易错点：

- follower 不是简单把 leader 日志直接拼上去，而是先检查前一条日志是否匹配。
- 冲突日志如果未提交，可以被新 leader 覆盖。
- 已提交日志不能被覆盖，这是 Raft 安全性的核心。

#### 8.5.4 Leader 崩溃场景

常见题型：

```text
Leader 收到客户端写入后宕机。
该日志只复制到少数派，或已经复制到多数派，结果有什么不同？
```

答题模板：

```text
如果日志尚未复制到多数派，leader 崩溃后该日志不能认为已经提交。
新 leader 当选后，未提交日志可能被覆盖，客户端需要重试。
如果日志已经复制到多数派，并满足提交条件，则该日志会被保留下来。
因为之后能当选的新 leader 必须获得多数派投票，而任意两个多数派有交集；
投票时的日志新旧限制会阻止缺少已提交日志的节点当选。
```

判断口诀：

```text
少数派复制：不算提交，可能丢。
多数派复制并提交：必须保留，不能丢。
```

更严谨的补充：

```text
Raft 对当前任期日志的提交最直接：当前 term 的日志复制到多数派后，leader 可提交它，并间接提交之前未提交的旧 term 日志。
旧 term 的日志即使存放在多数派上，也通常要等当前 term 的新日志提交后一起被安全提交。
考试若没有深入到旧 term 细节，抓“多数派 + leader 提交 + 新 leader 保留”即可。
```

#### 8.5.5 网络分区和脑裂场景

常见题型：

```text
5 节点集群分成 2/3 两个分区。旧 leader 在 2 节点分区里，还能继续提交吗？
另一个 3 节点分区会发生什么？
```

答题模板：

```text
Raft 依赖多数派避免脑裂。
旧 leader 如果处于少数派分区，即使还能接收客户端请求，也无法把日志复制到多数派，因此不能提交新写。
多数派分区中的节点会在超时后发起选举，获得多数票后产生新 leader。
当旧 leader 之后收到更高 term 的消息时，必须退回 follower。
少数派期间产生的未提交日志会被新 leader 的日志覆盖。
```

易错点：

- 少数派旧 leader “能追加本地日志”不等于“能提交”。
- Raft 牺牲少数派可用性来保证一致性。
- 脑裂避免靠多数派和 term，而不是靠所有节点同时在线。

#### 8.5.6 日志新旧和投票限制场景

常见题型：

```text
一个日志较旧的节点发起选举，能否成为 leader？
```

答题模板：

```text
Raft 投票时不仅看 term，还看候选者日志是否足够新。
比较规则是：先比较最后一条日志的 term，term 更大者更新；
如果最后日志 term 相同，再比较 log index，index 更大者更新。
投票者只会把票投给日志至少和自己一样新的 candidate。
这个限制保证已提交日志不会被缺少该日志的新 leader 覆盖。
```

易错点：

- term 高不代表日志一定新；投票仍要检查日志新旧。
- 这个规则是 Raft 安全性比“单纯选主”多出来的关键点。

#### 8.5.7 高频押题

1. `5` 节点集群 leader 崩溃，两个 candidate 同时选举，问谁能当 leader、分票怎么办。
2. leader 写入一条日志，只复制到 `2/5` 节点就宕机，问这条日志是否提交、会不会丢。
3. leader 写入一条日志，复制到 `3/5` 节点后宕机，问新 leader 是否必须保留。
4. follower 日志和 leader 不一致，给出两边日志表，问 AppendEntries 如何回退和覆盖冲突日志。
5. 网络分区为 `2 + 3`，旧 leader 在少数派，问能不能继续提交，恢复后会发生什么。
6. 日志较旧的节点发起选举，问为什么可能拿不到票，以及日志新旧如何比较。
7. 结合 2PC 对比 Raft：2PC 协调者故障为什么阻塞，Raft 为什么多数派存活仍能继续。

## 9. 2PC vs Paxos/Raft

| 项目 | 2PC | Paxos/Raft |
| --- | --- | --- |
| 解决问题 | 原子提交 | 共识/复制状态机 |
| 核心角色 | 协调者 + 参与者 | 多数派节点，Raft 有 leader |
| 故障风险 | 协调者故障可阻塞 | 多数派存活即可继续 |
| 典型用途 | 分布式事务提交 | 元数据、配置、日志复制 |

标准答案：

```text
2PC 解决多个参与者对提交或中止结果保持一致的问题，重点是分布式事务原子提交。
Paxos/Raft 解决多个节点对某个值或日志顺序达成共识的问题，常用于复制状态机。
2PC 依赖协调者，协调者故障时可能阻塞；Paxos/Raft 依赖多数派，只要多数派可通信，通常可以继续选主和提交。
```

## 9.5 进程组、可靠组播和原子组播

PPT 的容错章节会把“进程组 + 组播”作为容错基础。

| 概念 | 含义 |
| --- | --- |
| Flat group | 组内成员地位平等，没有固定协调者 |
| Hierarchical group | 有 coordinator，组内操作由协调者管理 |
| Reliable multicast | 消息若交付给一个正确成员，最终也应交付给所有正确成员 |
| Atomic multicast | 在可靠组播基础上，所有正确成员以相同顺序交付消息 |
| Virtual synchrony | 组成员视图变化和消息交付保持一致，避免成员对“谁在组内”理解不同 |

```text
可靠组播关注消息是否到达所有非故障成员；
原子组播进一步要求所有成员按相同顺序交付消息，因此可用于主动复制和复制状态机。
虚拟同步把成员加入、离开、故障检测与消息交付绑定在一起，让进程组在视图切换时仍保持一致语义。
```

知识点展开：

```text
进程组是把多个进程作为一个逻辑整体来管理的容错手段。
flat group 中成员地位平等，任何成员都可参与决策，优点是没有固定中心，缺点是协调成本高；
hierarchical group 中有 coordinator 负责管理组内操作，优点是控制简单，缺点是 coordinator 可能成为瓶颈或单点故障。
```

组播语义对比：

| 语义 | 保证 | 不保证 |
| --- | --- | --- |
| Basic multicast | 尽力发送给多个成员 | 不保证每个正确成员都收到 |
| Reliable multicast | 一个正确成员交付后，所有正确成员最终交付 | 不一定保证交付顺序相同 |
| Atomic multicast | 所有正确成员交付同一组消息，并按相同顺序交付 | 代价更高，通常要排序/共识 |

答题模板：

```text
可靠组播用于保证组内消息不会只被部分正确成员看到。
如果某条消息被一个正确成员交付，那么所有正确成员最终也应交付该消息。
原子组播在可靠组播基础上增加全序要求：所有正确成员必须以相同顺序交付消息。
因此原子组播可用于主动复制，因为所有副本按同一顺序执行同一请求时，状态才能保持一致。
虚拟同步进一步把成员视图变化和消息交付联系起来，保证进程在同一组视图下对消息集合有一致理解。
```

易错点：

- 可靠组播解决“到没到所有正确成员”，原子组播解决“大家交付顺序是否一致”。
- 原子组播比因果组播更强：原子组播要求全序，因果组播只约束有因果关系的消息。

## 10. 容灾与脑裂

标准答案：

```text
容灾是指系统在机器、机房、网络或地域故障时，仍尽量保证服务可用和数据不丢失。
常见方式包括备份、主备切换、同城双活、异地多活和数据复制。
脑裂是网络分区后多个节点或机房都认为自己是主节点，可能同时接受写入，导致数据冲突。
多数派机制可通过只允许获得多数派的一侧继续服务来避免脑裂，但会牺牲少数派分区的可用性。
```

## 11. 常见大题模板

### 11.1 `2K+1` 和 `3K+1`

```text
这两个公式必须结合故障模型理解。
如果错误节点只是返回任意值，而正确节点结果唯一且相同，可以用多数投票，容忍 K 个错误节点至少需要 2K+1 个副本。
如果错误节点可能向不同节点发送不同信息，属于拜占庭故障，并要求所有正确节点达成一致，则常见要求至少 3K+1 个节点。
```

### 11.2 Raft 为什么依赖多数派

```text
Raft 依赖多数派是为了保证不同决策之间存在交集。
Leader 需要多数票才能当选，日志需要复制到多数派才能提交。
任意两个多数派必然有交集，因此新 leader 能通过交集节点继承已提交日志，避免两个 leader 在不同分区提交冲突日志。
这可以避免脑裂，但少数派分区无法继续提交写请求。
```

## 12. 考前最小背诵清单

- 可信系统：availability、reliability、safety、maintainability。
- 提高可信性：故障预防、故障容忍、故障移除、故障预测。
- 故障模型：crash、omission、timing、response、arbitrary/Byzantine。
- 冗余：information、time、physical。
- `K+1`：fail-silent/crash 后至少剩一个正确副本。
- `2K+1`：唯一正确结果 + 多数投票。
- `3K+1`：拜占庭故障 + 正确节点一致。
- 回退恢复：回到 checkpoint 后重新执行。
- 前向恢复：从错误状态修复到正确状态。
- checkpoint：保存一致状态，限制恢复范围。
- checkpoint 方式：independent、coordinated、two-phase。
- 2PC：投票阶段、决定阶段。
- 2PC 阻塞：参与者 yes 后协调者宕机，参与者不知道 commit/abort。
- 2PC ready/prepared 状态不能单方面决定。
- 3PC：CanCommit、PreCommit、DoCommit，减少但不绝对消除阻塞。
- 组播：reliable multicast 保证到达；atomic multicast 保证相同顺序交付。
- Paxos：prepare/promise，accept/accepted，proposal number，多数派交集。属于往年题补充。
- Raft 三部分：leader 选举、日志复制、安全性。属于往年题补充。
- Raft 核心：term、log、majority，已提交日志必须被新 leader 保留。
