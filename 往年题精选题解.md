## 一、分布式系统模型与架构

### 1. 什么是分布式系统？为什么要使用分布式系统？

来源：2022；最新 PDF 明确主线。

为什么选：

- 定义题几乎是基本盘。
- 容易漏写“对用户像一个系统”。

题目：

> 分布式系统的定义是什么？为什么要使用分布式系统？写出至少 3 点原因。

答案：

```text
分布式系统是由多个自治计算元素组成的系统，这些计算元素通过网络通信和协作，对用户表现为一个统一、连贯的系统。

使用分布式系统的原因包括：
1. 资源共享：让用户访问远程数据、设备和服务。
2. 性能提升：多节点并行处理，提高吞吐量和计算能力。
3. 可靠性：部分节点故障时，系统可通过冗余继续服务。
4. 固有分布性：用户、数据或设备本来就分布在不同地点。
5. 可扩展性：可通过增加节点逐步扩容。
6. 经济性：多台普通机器通常比一台大型机更便宜。
```

易错点：

- 不能只写“多台机器联网”，必须写“对用户表现为一个统一系统”。
- “为什么分布式”不要只写性能，可靠性、资源共享、固有分布性也常给分。

### 2. 分布式系统的目标有哪些？

来源：最新 PDF。

题目：

> 分布式系统的主要目标是什么？

答案：

```text
分布式系统的目标包括资源可用、分布透明性、开放性和可扩展性。
资源可用指让远程资源、数据和服务可被共享访问；
分布透明性指隐藏访问、位置、复制、迁移、并发和故障等分布式细节；
开放性指通过标准接口和协议支持互操作、可移植和扩展；
可扩展性指系统在节点数、用户数和地域范围扩大时仍能有效工作。
```

易错点：

- “目标”和“原因”有交叉，但目标更偏资源共享、透明性、开放性、可扩展性。

### 3. 如何理解分布式系统的透明性？

来源：2021；最新 PDF 明确主线。

题目：

> 如何理解分布式系统的透明性？列举常见透明性类型。

答案：

```text
透明性是指分布式系统隐藏资源分布和实现细节，使用户像使用一个统一系统一样使用分布式资源。
常见透明性包括：
1. 访问透明性：隐藏本地和远程访问方式差异。
2. 位置透明性：隐藏资源物理位置。
3. 迁移透明性：资源迁移不影响访问方式。
4. 重定位透明性：资源使用过程中移动也不影响访问。
5. 复制透明性：隐藏多个副本的存在。
6. 并发透明性：并发访问不破坏共享资源一致状态。
7. 故障透明性：尽量隐藏或恢复部分故障。
```

易错点：

- 故障透明性不是“永不失败”，而是尽量屏蔽或恢复故障。
- 透明性不是越强越好，过度透明会带来实现复杂度和性能成本。

### 4. 分布式系统的开放性体现在哪些方面？

来源：2019；最新 PDF 明确主线。

题目：

> 分布式系统的开放性主要体现在哪些方面？

答案：

```text
开放性主要体现在接口和协议的规范化。
开放系统应能与其他开放系统服务交互，而不依赖特定硬件、操作系统或网络环境。
它要求系统具有定义良好的接口，支持应用可移植性，容易互操作，并尽量屏蔽底层环境的异构性。
开放性使系统更容易替换组件、接入新服务和扩展功能。
```

易错点：

- 开放性不是“所有资源公开访问”，而是“接口规范、互操作、可移植、可扩展”。

### 5. 分布式操作系统、网络操作系统和基于中间件的系统有什么区别？

来源：2020；最新 PDF 明确主线。

题目：

> 网络操作系统和基于中间件的分布式系统分别是什么？它们和分布式操作系统有什么区别？

答案：

```text
分布式操作系统试图在操作系统层统一管理多台机器，使多台机器尽量像单机一样工作。
网络操作系统通过网络连接多台计算机，提供远程登录、文件共享等功能，但用户通常仍能感知资源位于哪台机器。
基于中间件的分布式系统在操作系统和应用之间加入中间层，通过统一接口、命名、通信和服务发现屏蔽异构性，使多个独立节点对应用表现为统一系统。
```

易错点：

- 网络操作系统不等于分布式系统，它通常缺少充分的单一系统映像。
- 中间件不是操作系统，而是 OS 和应用之间的分布式抽象层。

### 6. 策略和机制如何区分？

来源：2015、2020、2019；高频判断题。

题目：

> 判断以下是策略还是机制，并给出对应机制或策略：根据应用需求动态设定缓存大小；支持不同类型加密系统；能够通过多条路径传递数据包。

答案：

```text
策略决定“做什么”，机制提供“怎么做”。

1. 根据应用需求动态设定缓存大小：策略。
   对应机制：系统支持动态调整缓存容量。
2. 支持不同类型加密系统：机制。
   对应策略：根据安全等级或性能要求选择某种加密算法。
3. 能够通过多条路径传递数据包：机制。
   对应策略：根据拥塞程度、延迟或可靠性选择最合适路径。
```

易错点：

- 判断时先问：这是选择规则，还是系统提供的能力？
- “按什么条件选择”通常是策略；“支持这种能力”通常是机制。

常见判断表：

| 名词 | 更像策略还是机制 | 对应另一侧 |
| --- | --- | --- |
| 最近节点优先 | 策略 | 服务发现、延迟探测、路由 |
| 请求超时后重试 3 次 | 策略 | timer、请求 ID、重传 |
| 负载均衡器 | 机制 | 轮询、最少连接、最低延迟策略 |
| 心跳检测 | 机制 | 故障切换策略 |

### 7. BitTorrent 是什么架构？基本工作方式是什么？

来源：2019。

题目：

> BitTorrent 是一种内容分发协议。简述基于该协议的分布式内容共享系统基本架构和工作方式。

答案：

```text
BitTorrent 可看作混合式架构。
它通常用集中式目录或 tracker 帮助节点发现资源和其他 peer，
但真正的数据文件传输在 peer 之间以 P2P 方式完成。
文件会被切成多个块，不同 peer 可并行交换不同块，从而减轻中心服务器压力并提升下载吞吐量。
```

易错点：

- 不要写成完全中心化；目录/发现可中心化，数据传输是 P2P。

### 8. 集中式、非集中式和混合式组织有什么区别？

来源：Peanuts 补充；最新 PDF 架构组织形式。

题目：

> 简述集中式、非集中式和混合式分布式组织的特点、优缺点，并举例。

答案：

```text
集中式组织依赖中心服务器或中心协调者，结构简单、易管理，但中心节点可能成为瓶颈和单点故障。
非集中式组织没有固定中心，节点对等协作，扩展性和鲁棒性较好，但查找、路由和一致性维护更复杂。
混合式组织结合二者，例如中心节点负责索引或发现，实际数据由 peer 之间直接传输。
BitTorrent 可视为混合式系统：tracker 或 DHT 帮助发现 peer，文件块在 peer 间交换。
```

易错点：

- 不要把“有 tracker”理解为完全中心化，关键看数据路径和控制路径是否分离。
- 非集中式不是一定更好，它把单点问题换成了查找、路由和一致性维护成本。

### 9. 结构化 P2P 和非结构化 P2P 有什么区别？

来源：Peanuts 补充；P2P 架构题。

题目：

> 比较结构化 P2P 和非结构化 P2P 的组织方式、查找方式和优缺点。

答案：

```text
结构化 P2P 使用 DHT、逻辑环或超立方体等覆盖网络结构组织节点，使资源能按 key 高效定位。
优点是查找较确定、高效、可扩展；缺点是节点频繁加入离开时维护成本高。

非结构化 P2P 中节点通常随机选择邻居，不维护严格全局结构，常通过泛洪或随机游走查找资源。
优点是简单、灵活，适合动态环境；缺点是查找不确定、消息开销大，负载可能不均衡。
```

易错点：

- 结构化 P2P 的“结构”指覆盖网络的逻辑结构，不是物理网络拓扑。
- 非结构化 P2P 简单但不等于低开销；泛洪查找可能产生大量消息。

### 10. 进程和线程的异同？Web 应用采用多线程有什么优势？

来源：2019；最新 PDF 明确主线。

题目：

> 简述进程和线程的异同，一个 Web 应用采用多线程结构有哪些优势？

答案：

```text
进程和线程都是程序执行单元，都可被操作系统调度并并发执行。
进程拥有独立地址空间和资源，隔离性强但创建和切换成本较高；
线程是轻量级进程，同一进程内多个线程共享地址空间和资源，切换成本较低但隔离性较弱。

Web 应用采用多线程的优势包括：
1. 可并发处理多个客户端请求，提高吞吐量。
2. 一个线程等待 I/O 时，其他线程仍可继续执行，降低响应时间。
3. 同一进程内线程共享缓存、连接池等资源，通信成本较低。
```

易错点：

- 线程共享同一进程地址空间；进程之间默认不共享地址空间。
- 线程崩溃可能影响整个进程。

### 11. 强迁移和弱迁移分别是什么？

来源：2022；最新 PDF 明确主线。

题目：

> 代码迁移中的强迁移和弱迁移分别是什么？

答案：

```text
强迁移会迁移代码、数据以及执行状态，使运行实体可以在目标机器上从原执行点继续运行。
弱迁移只迁移代码和必要数据，迁移后通常从预定义入口重新开始执行。
强迁移透明性更好，但实现复杂；弱迁移实现简单，但对程序结构和恢复逻辑有更多要求。
```

易错点：

- 强迁移的关键是执行状态；弱迁移没有完整执行状态。

### 12. 代码迁移和虚拟机迁移有什么区别？

来源：Peanuts 补充。

题目：

> 比较代码迁移和虚拟机迁移的迁移对象、适用场景和优缺点。

答案：

```text
代码迁移迁移的是程序代码和必要数据；强迁移还包含执行状态。
它粒度较小、灵活，但要求目标环境兼容，安全性和依赖管理更复杂。

虚拟机迁移迁移的是完整虚拟机运行环境，包括操作系统、进程、内存状态、磁盘状态和网络连接等。
它对应用更透明，适合云平台负载均衡、维护和故障迁移；
缺点是迁移数据量更大，内存、磁盘、网络连接和生命周期管理更复杂，并有一定性能开销。
```

易错点：

- 代码迁移关注程序实体；虚拟机迁移关注完整运行环境。
- 虚拟机迁移更透明，但不是更轻量。

## 二、通信与 RPC

### 13. TCP 和 UDP 位于哪一层？用于 RPC 各有什么好处？

来源：2022。

题目：

> TCP 和 UDP 分别位于网络堆栈哪一层？使用 TCP 和 UDP 进行 RPC 通信分别有什么好处？

答案：

```text
TCP 和 UDP 都位于传输层。
TCP 是面向连接的可靠字节流协议，提供按序、可靠传输，并带有流量控制和拥塞控制，适合对可靠性要求高的 RPC。
UDP 是无连接协议，开销小、延迟低，不保证可靠和顺序，适合实时性强、允许丢包或由应用层自行实现可靠性的场景。
```

易错点：

- TCP/UDP 是传输层协议；同步/异步是应用层调用方式。

### 14. “文本消息程序应该使用 UDP 传输”是否正确？

来源：2022 判断题。

题目：

> 判断：“文本消息程序应该使用 UDP 进行传输。”并说明理由。

答案：

```text
不一定，通常不应简单使用裸 UDP。
文本消息通常要求可靠送达、顺序一致和不重复，TCP 能直接提供可靠、有序传输。
UDP 开销低、延迟小，但不保证消息到达和顺序，若使用 UDP，则应用层必须额外实现确认、重传、去重和排序。
因此该说法过于绝对。
```

易错点：

- 不要写“UDP 一定不能用”。如果应用层自己补可靠性，也可以用 UDP。

### 15. 有了 RPC，进程能否使用自己的本地内存地址执行远程程序？

来源：2022 判断题。

题目：

> 判断：“有了 RPC，进程就可以使用自己本地的内存地址执行远程程序。”并说明理由。

答案：

```text
错误。
RPC 只是把远程过程调用包装得像本地调用，通过 stub 进行参数编组、网络传输和结果解组。
客户端和服务器处于不同地址空间，客户端本地内存地址在服务器进程中没有意义。
RPC 传递的是参数值或可序列化对象，而不是让远程进程直接访问调用方本地地址。
```

易错点：

- “像本地调用”只是调用形式，不代表共享内存地址空间。

### 16. 同步通信和异步通信是什么？TCP 是同步还是异步？

来源：2021；2024 标红易错。

题目：

> 简述同步通信和异步通信含义，一个 TCP 连接中数据传输是同步还是异步？为什么？

答案：

```text
同步通信中，发送方发送请求后需要等待接收方响应或通信完成，才能继续后续逻辑。
异步通信中，发送方发送请求后可以继续执行，不必立即等待响应，结果可通过回调、通知、轮询或消息队列返回。

TCP 本身不是同步或异步通信的分类对象。TCP 是可靠、面向连接的字节流传输协议；
同步/异步描述的是应用层调用方式。在阻塞式 socket 或同步 RPC 中，TCP 通信可表现为调用方等待响应；
在非阻塞 I/O 或事件驱动程序中，也可以基于 TCP 实现异步通信。
```

易错点：

- 不要写“TCP 是同步”。更严谨：TCP 是传输层协议，同步/异步取决于应用层使用方式。

### 17. 持久通信和非持久通信的区别？电子邮件和 QQ 留言属于哪种？

来源：2021、2019、2020。

题目：

> 持久通信和非持久通信有什么区别？电子邮件和 QQ 留言采用什么通信模式？如何基于 TCP/IP 实现？

答案：

```text
瞬时/非持久通信要求发送和接收进程在通信期间同时运行，通信系统不会长期保存消息。
持久通信中，消息提交后由通信系统或中间件保存，直到接收方可接收或消息过期。

电子邮件属于持久通信。发送方先通过 TCP/IP 将邮件提交给邮件服务器，邮件服务器保存邮件；
接收方之后再通过 TCP/IP 从服务器拉取或接收邮件。即使接收方暂时不在线，邮件也不会立即丢失。

QQ 留言也可按持久通信理解：留言先由服务器保存，接收方之后上线再读取。
```

易错点：

- 持久通信的核心是“通信系统保存消息”，不是“发送连接一直保持”。

### 18. RPC 的基本工作过程是什么？

来源：Peanuts 补充；RPC 基础口径。

题目：

> 简述一次 RPC 调用从客户端到服务器再返回的基本过程。

答案：

```text
RPC 通过 client stub 和 server stub 隐藏网络通信细节。
客户端像调用本地过程一样调用 client stub；
client stub 将过程名、参数和请求 ID 等信息编组为消息，通过网络发送给服务器；
server stub 接收消息并解组参数，然后调用服务器端真实过程；
服务器执行后把结果交给 server stub；
server stub 编组响应并返回客户端；
client stub 解组响应，把结果返回给调用方。
```

易错点：

- “像本地调用”只是调用形式，客户端和服务器仍在不同地址空间。
- stub 负责编组、解组和网络通信包装，不负责业务逻辑本身。

### 19. 如何在 TCP 基础上实现 RPC 的 at-most-once？

来源：2019、2020。

题目：

> 简述如何在 TCP 基础上实现远程过程调用 RPC 的“最多执行一次”机制。

答案：

```text
at-most-once 的目标是同一个 RPC 请求最多被执行一次。
常见做法是：
1. 客户端为每个请求生成唯一请求 ID。
2. 服务器维护已处理请求表，记录请求 ID 和结果。
3. 第一次收到请求时执行，并缓存结果。
4. 若收到相同 ID 的重复请求，不重新执行，而是返回缓存结果或直接说明已处理。
5. 已处理请求表需要设置过期策略，避免无限增长。
```

易错点：

- TCP 可靠传输不等于 RPC 不会重复执行。
- 响应丢失后客户端重试，服务器若无去重仍可能重复执行。

### 20. 动态绑定为什么需要？缺点是什么？

来源：Peanuts 补充；RPC 服务发现题。

题目：

> RPC 或分布式服务为什么需要动态绑定？动态绑定有什么缺点？

答案：

```text
动态绑定让客户端在运行时通过 binder、注册中心或服务发现机制查找服务实例，而不是写死服务器地址。
它支持服务迁移、扩容、多版本、负载均衡、容错切换和接口版本匹配。

缺点是增加注册和查询开销；binder 或注册中心可能成为性能瓶颈或单点故障；
客户端还要处理缓存失效、服务实例变化和版本兼容问题。
因此动态绑定本身也需要缓存、复制、高可用和健康检查机制。
```

易错点：

- 动态绑定不是“运行时修改程序”，而是运行时解析服务位置或服务实例。
- 注册中心提高灵活性，但它自己也要做高可用。

### 21. 客户端没有收到服务器响应，可能有哪些情况？如何处理？

来源：2020；RPC 故障高频。

题目：

> 客户端没有收到来自服务器的响应，可能的情况有哪些？并给出可能解决方案。

答案：

```text
可能情况包括：
1. 请求消息没有到达服务器。
2. 请求已到达，但服务器处理很慢。
3. 服务器接收请求后在执行前、执行中或执行后崩溃。
4. 服务器已执行成功，但响应消息丢失。
5. 客户端无法定位服务器、连接失败，或等待超时设置过短。

处理方法：
1. 设置超时并重试。
2. 请求携带唯一 ID，服务器做去重和结果缓存。
3. 对有副作用操作使用幂等键、事务日志或状态查询。
4. 对服务器故障使用恢复日志和补偿机制。
5. 对扣费、下单等业务操作使用业务流水号和端到端确认，避免重试造成重复副作用。
```

易错点：

- 超时不等于失败；超时只表示客户端没按时收到结果。
- 对无副作用或幂等操作可直接重试；对有副作用操作必须先设计去重和幂等。

### 22. RPC 中客户端崩溃如何处理？

来源：2021；2024 标红。

题目：

> RPC 中客户端发送请求后崩溃会出现什么问题？处理方法有哪些？

答案：

```text
客户端崩溃后，服务器可能仍在执行该客户端发起的远程计算，形成 orphan computation（孤儿计算）。
处理方法包括：
1. Extermination：客户端调用前记录日志，恢复后根据日志清理孤儿。
2. Reincarnation：客户端重启后广播新的 epoch，杀死旧 epoch 相关远程计算。
3. Gentle reincarnation：收到新 epoch 后先检查计算是否仍有 owner，无 owner 再终止。
4. Expiration：为 RPC 设置租约或超时时间，到期未续约则自动结束。
```

易错点：

- 客户端崩溃问题重点不是“请求丢失”，而是服务器端可能继续执行无人负责的计算。

### 23. 限制提交次数的 RPC 如何体现 at-least-once？如何丢弃重复子任务？

来源：2022。

题目：

> RPC 代码只发送一次请求并等待 ACK，要求体现 at-least-once 语义但限制提交次数。为什么原代码没有体现 at-least-once？如何修改？包中应包含哪些信息使服务器丢弃重复子任务？

答案：

```text
原代码只发送一次请求，如果请求或响应丢失就不再重试，因此不能保证 at-least-once。
应修改为：客户端设置超时，在未收到 ACK 时重发请求，直到收到 ACK 或达到提交次数上限。

为避免重试导致重复执行，请求包中应包含：
1. 客户端 ID。
2. 请求/任务唯一 ID。
3. 子任务编号或序列号。
4. 必要的重试轮次或时间戳。

服务器根据客户端 ID + 请求 ID / 子任务 ID 判断是否重复；若已处理，则丢弃重复执行或返回缓存结果。
```

易错点：

- at-least-once 依赖重试。
- 丢弃重复任务依赖唯一标识和服务端去重。

### 24. “单用户只能 7 次查重”的 RPC 如何避免重复扣次数？

来源：Peanuts 补充；RPC 幂等和业务次数限制。

题目：

> 某查重服务规定单用户最多查重 7 次。若客户端可能因响应丢失而重试，如何设计 RPC，避免一次业务请求被重复扣次数？

答案：

```text
客户端每次业务请求携带用户 ID 和全局唯一请求 ID。
服务器在持久化存储中维护用户已成功查重次数，以及已处理请求 ID 到结果的映射。

收到请求后，服务器先检查请求 ID 是否已处理：
若已处理，直接返回缓存结果，不重复执行，也不重复扣次数。
若是新请求，则在事务中检查该用户成功次数是否小于 7。
若小于 7，执行查重、次数加 1，并记录请求 ID 和结果；
若已经达到 7 次，则拒绝请求。

这样即使响应丢失导致客户端重试，服务端也能通过请求 ID 实现幂等效果，避免重复扣业务次数。
```

易错点：

- 只靠 TCP 不够；业务层必须有请求 ID、去重表、结果缓存和事务更新。
- “最多执行一次”要落到业务效果上：不能重复扣次数。

### 25. 稳定 WiFi 和开放系统中的丢包如何改进可靠性？

来源：Peanuts 补充；可靠通信题。

题目：

> 在稳定 WiFi 或开放系统中出现丢包时，如何从协议和应用层改进可靠性？

答案：

```text
传输层可通过 ACK/NAK、超时重传、序列号、校验和、滑动窗口实现可靠传输。
发送方为分组编号，接收方确认收到的分组；
发送方超时未收到确认则重传；
接收方利用序列号去重、重排并检测缺失。

实时流媒体中，完全重传可能增加延迟，可使用缓冲、交叉传输、冗余包或 FEC 分散并恢复部分丢包。
开放系统中还应在应用层加入幂等设计、请求去重、持久消息队列和端到端确认，避免网络可靠但业务重复或丢失。
```

易错点：

- 可靠传输和业务可靠不是一回事；开放系统常需要端到端确认和幂等。
- 实时场景不能只答“全部重传”，还要考虑延迟和抖动。

## 三、时间同步与逻辑时钟

### 26. 为什么分布式系统不适合用物理时钟判断顺序？逻辑时钟是什么？

来源：2019、2021；最新 PDF 主线。

题目：

> 为什么在分布式系统中不适合用物理时钟进行同步？简述逻辑时钟的基本含义。

答案：

```text
分布式系统中每台机器有自己的物理时钟，不同机器会产生时钟漂移。
网络传输延迟也不确定，因此仅靠物理时间无法可靠判断事件的因果先后。
即使使用 NTP 等算法，也只能减小误差，不能彻底消除分布式事件排序问题。

逻辑时钟不追求真实墙上时间，而是给事件分配逻辑时间，用于刻画事件之间的相对顺序和因果关系。
在许多分布式算法中，重要的是进程对事件顺序达成一致，而不是所有时钟完全等于真实时间。
```

易错点：

- 不要说“物理时钟完全不能同步”，应说“不能可靠提供因果顺序”。

### 27. 设计平均时间同步协议

来源：2021、2020。

题目：

> 设计一个协议，使同步后时间为所有节点时间的平均值，写出同步和时间调整流程。

答案：

```text
一种简单平均时间同步协议：
1. 每个节点向其他所有节点发送自己的本地时间戳。
2. 每个节点收集其他节点返回的时间值，并记录消息往返时间。
3. 可根据 RTT/2 对收到的时间做延迟估计修正。
4. 丢弃明显异常值或超时响应。
5. 对剩余时间取平均值或加权平均值。
6. 节点将本地时钟逐步调整到该平均值，避免时间突然回拨。

该协议不依赖单一时间服务器，但精度受网络延迟和异常节点影响。
```

易错点：

- 最好写“逐步调整”，不要让时钟突然大幅回退。

### 28. 分布式互斥：4 个进程如何尽可能快，并处理节点故障？

来源：Peanuts 补充；最新 PDF 同步主线。

题目：

> 4 个进程需要互斥访问临界区。如何设计协议使正常情况下尽可能快？如果节点故障如何处理？

答案：

```text
若追求正常情况下消息少和速度快，可选集中式互斥。
选一个 coordinator 管理临界区。
进程进入临界区前向 coordinator 请求许可；
coordinator 空闲时授权，否则把请求排队；
进程退出临界区后通知 coordinator 释放。

正常情况下进入一次临界区约需 request、grant、release 三条消息。
如果普通请求节点故障，不影响其他节点继续向 coordinator 申请；
如果 coordinator 故障，需要通过选举或备份 coordinator 产生新的协调者，并恢复等待队列和锁状态。
```

易错点：

- 只说“集中式最快”不够，要补 coordinator 故障时需要选举或备份，否则它本身是单点故障。
- 如果题目更强调无单点故障，可考虑分布式互斥算法，但消息数通常更多。

### 29. Cristian 算法计算题

来源：2022。

题目：

> 客户端在 11:45:14.740 发起请求，11:45:15.810 收到响应。响应中服务器时间为 11:45:14.970。客户端估计服务器到客户端的单趟延迟至少为 0.400s。求客户端收到消息的最早真实时刻和最晚真实时刻。

答案：

```text
客户端收到响应时，服务器发出响应的真实时间为 11:45:14.970。
服务器到客户端单趟延迟至少为 0.400s，因此客户端收到响应的最早真实时刻为：
11:45:14.970 + 0.400s = 11:45:15.370。

客户端从发送到接收的本地经过时间为：
11:45:15.810 - 11:45:14.740 = 1.070s。
若服务器到客户端至少 0.400s，则客户端到服务器最多为 1.070 - 0.400 = 0.670s。
因此客户端收到响应的最晚真实时刻可由服务器发送时间加最大可能单趟延迟估计：
11:45:14.970 + 0.670s = 11:45:15.640。
```

易错点：

- 题目问“收到消息的真实时刻范围”，不是 Cristian 标准算法里直接设置的本地时间。
- 标准 Cristian 估计时间通常是 `server_time + RTT/2`，但这里给了最小单趟延迟，要算范围。

### 30. Lamport 时钟计算：给出消息序列

来源：2021。

题目：

> 3 个进程 A/B/C 初始逻辑时钟均为 0，事件依次为：A 向 B 发送 M1；A 向 C 发送 M2；A 向 C 发送 M3；C 先收到 M3，再收到 M2，然后向 B 发送 M4；B 收到 M4 后向 A 发送 M5；A 收到 M5 后向 C 发送 M6；C 收到 M6。写出每个消息 Lamport 时间戳和最后三个进程本地时间。

答案：

```text
Lamport 规则：本地/发送事件时 C=C+1；接收时间戳 t 的消息时 C=max(C,t)+1。

初始 A=B=C=0。
1. A 发送 M1：A=1，M1=1。
2. A 发送 M2：A=2，M2=2。
3. A 发送 M3：A=3，M3=3。
4. C 接收 M3：C=max(0,3)+1=4。
5. C 接收 M2：C=max(4,2)+1=5。
6. C 发送 M4：C=6，M4=6。
7. B 接收 M1：B=max(0,1)+1=2。
8. B 接收 M4：B=max(2,6)+1=7。
9. B 发送 M5：B=8，M5=8。
10. A 接收 M5：A=max(3,8)+1=9。
11. A 发送 M6：A=10，M6=10。
12. C 接收 M6：C=max(6,10)+1=11。

消息时间戳：
M1=1，M2=2，M3=3，M4=6，M5=8，M6=10。
最终本地时间：
A=10，B=8，C=11。

如果同题要求 M1-M4 的向量时间戳，按进程顺序 A/B/C 可写为：
M1=[1,0,0]，M2=[2,0,0]，M3=[3,0,0]，M4=[3,0,3]。
```

易错点：

- B 收到 M1 的具体位置题目未完全写明；如果按“B 收到 M4 后向 A 发 M5”之前必须已收到 M1 处理，上述是一种自然顺序。
- 接收 M2 时 C 已经因为 M3 到 4，所以 M2 不会把 C 拉回去。

### 31. Lamport 是全序还是偏序？向量时钟为什么能判断并发？

来源：2020、2021、2022。

题目：

> Lamport 时钟是全序关系还是偏序关系？为什么 Lamport 不能判断并发而向量时钟可以？

答案：

```text
Lamport 时钟刻画的是 happens-before 偏序关系。
它满足：如果 a -> b，则 C(a) < C(b)。
但反过来不成立，C(a) < C(b) 不能推出 a -> b，因为两个并发事件也可能被赋予不同整数时间戳。
因此 Lamport 时间戳不能可靠判断两个事件是否并发。

向量时钟为每个进程维护一个分量。
若 V(a) 每一维都 <= V(b)，且至少一维 <，则 a -> b；
若 V(b) 每一维都 <= V(a)，且至少一维 <，则 b -> a；
否则二者并发。
所以向量时钟可以通过逐维比较判断因果关系和并发关系。
```

易错点：

- “Lamport + 进程 ID”可以构造人为全序，但不是因果全序。
- 向量时钟也不是为了构造全序，核心是判断并发。

## 四、一致性与复制

### 32. 顺序一致性、FIFO 一致性和因果一致性如何判断？

来源：2021、2022、2019。

题目：

> 给出若干读写执行，判断是否满足顺序一致性、FIFO 一致性或因果一致性，并说明理由。

答案模板：

```text
顺序一致性：
判断是否能找到一个所有进程都认可的全局操作顺序，并且该顺序保留每个进程内部程序顺序。

FIFO 一致性：
只要求同一进程发出的写操作被所有进程按发出顺序看到。
不同进程发出的写操作可以被不同进程以不同顺序看到。

因果一致性：
有因果关系的写操作必须被所有进程按相同因果顺序看到；
并发写操作可以被不同进程以不同顺序看到。
```

典型例子：

```text
若 P1 先 W(x=1) 再 W(x=2)，某进程看到 x=2 后又看到 x=1，则违反 FIFO，也违反因果和顺序。

若 P1 写 x=1，P2 并发写 x=2，P3 看到 1 后 2，P4 看到 2 后 1，
这可满足 FIFO 和因果一致性，但不满足顺序一致性，因为不存在所有进程共同认可的全局写顺序。

若 P2 读到 P1 的写 A 后再写 B，某进程看到 B 后看到 A，则违反因果一致性。
```

易错点：

- FIFO 只约束同一写进程的写顺序。
- 因果一致性允许并发写乱序。
- 顺序一致性不要求真实时间顺序，但要求所有进程能解释为同一个全局顺序。

### 33. 两个进程互相 kill：顺序一致性和 FIFO 一致性如何判断？

来源：Peanuts Q12；一致性典型易错题。

题目：

```text
初始 x = 0, y = 0

P1: x = 1
    if (y == 0) kill(P2)

P2: y = 1
    if (x == 0) kill(P1)
```

> 在顺序一致性和 FIFO 一致性下，是否可能出现两个进程互相 kill？

答案：

```text
记：
A = W1(x=1), B = R1(y)
C = W2(y=1), D = R2(x)

程序顺序要求：A < B, C < D。

顺序一致性要求存在一个所有进程共同认可的全局顺序，并保留每个进程内部程序顺序。
如果 P1 kill P2，则 B 读到 y=0，所以 B < C。
如果 P2 kill P1，则 D 读到 x=0，所以 D < A。
若两个进程互相 kill，则得到 A < B < C < D < A，形成环，不可能。
所以顺序一致性下双杀不可能；可能结果是最多一个进程被 kill，或者两个都不被 kill。

FIFO 一致性只要求同一写进程的写按顺序被所有进程看到。
本题中 P1 只有一个写 x=1，P2 只有一个写 y=1，
因此 FIFO 不要求 x=1 和 y=1 在所有进程看来有相同全局顺序。
可以出现 P1 暂时没看到 y=1，读 y=0 后 kill P2；
同时 P2 暂时没看到 x=1，读 x=0 后 kill P1。
这个“双杀”满足 FIFO，但不满足顺序一致性。
```

易错点：

- FIFO 不是“每次读到最近写入的数据”，而是“同一写进程的写顺序不被打乱”。
- 如果写成单一全局执行序列，就已经在用顺序一致性的思路，不能拿来证明 FIFO。

### 34. 数据为中心一致性和客户为中心一致性有什么异同？门户网站采用哪种？

来源：2019、2021；高频。

题目：

> 以数据为中心的一致性和以客户为中心的一致性有什么异同？若通过增加副本提升门户网站性能，副本应采用哪种一致性？

答案：

```text
数据为中心一致性从整个数据存储角度定义所有进程对共享数据读写应满足的顺序约束，
关注多个进程看到的数据操作是否一致，如顺序一致性、因果一致性等。

客户为中心一致性关注单个客户在访问不同副本时看到的结果是否符合自身预期，
典型保证包括单调读、单调写、读己之写和写后读。

门户网站通常读多写少，副本主要用于提升读取性能和降低访问延迟。
用户一般能容忍短时间旧数据，因此可采用客户为中心一致性或最终一致性。
具体要求是：没有新写入时副本最终收敛；同一用户尽量满足单调读和读己之写。
```

易错点：

- 不要只写“最终一致性即可”，必须解释读多写少、容忍旧数据、最终收敛。
- Paxos/Raft 是共识或复制状态机协议，不是“客户为中心一致性协议”。

### 35. Quorum-based 协议：N=10 时 R/W 如何取？

来源：2022、2019。

题目：

> Quorum-based 协议有 10 个节点，要保证读写不冲突、写写不冲突，请给出可能的写节点数量 W 和对应读节点数量 R 的范围。

答案：

```text
Quorum 条件：
1. R + W > N，保证读集合和写集合相交。
2. W > N/2，保证任意两个写集合相交。

N=10 时：
W > 5，所以 W 可取 6,7,8,9,10。
同时 R > 10 - W。

可行组合范围：
W=6，R>=5；
W=7，R>=4；
W=8，R>=3；
W=9，R>=2；
W=10，R>=1。
```

易错点：

- `R + W > N` 是严格大于。
- `W > N/2` 是写写冲突条件。

### 36. 复制的优势和不足是什么？

来源：最新 PDF 主线，常和一致性结合。

题目：

> 为什么要复制？为什么复制会带来问题？

答案：

```text
复制的优势：
1. 可靠性：避免单点故障。
2. 性能：多个副本分担访问压力，降低访问延迟。
3. 可扩展性：支持更多用户和更大地理范围。

复制的不足：
1. 复制透明性需要额外机制。
2. 多副本会产生一致性问题。
3. 更新需要传播到多个副本，代价更高。
4. 如果同步策略不当，可能降低可用性。
```

易错点：

- 复制不是只提升可用性，也会增加一致性和更新成本。

## 五、容错、恢复与分布式提交

### 37. K 容错唯一正确结果场景为什么需要 2K+1？

来源：2019、2020、2021、2022。

题目：

> 某非集中式系统中，正确进程必然返回唯一正确结果，错误进程可能沉默或返回随机错误值。K 容错至少需要多少副本？设计协议并说明原因。

答案：

```text
至少需要 2K+1 个副本。
因为最多 K 个副本出错时，仍有 K+1 个正确副本返回相同正确结果。
通过多数投票，正确结果的票数 K+1 会超过错误副本最多 K 票，因此可选出正确结果。

协议：
1. 客户端向全部 2K+1 个副本发送请求。
2. 收集副本返回结果，超时副本视为未响应。
3. 对返回值进行多数投票。
4. 出现超过 K 个相同结果时，将其作为正确结果返回。
```

易错点：

- 2K+1 适用于“正确结果唯一且相同”的场景。
- 如果错误节点可能向不同节点发送不同信息并要求正确节点一致，就要考虑更强拜占庭模型。

### 38. 传感器平均值场景为什么需要 3K+1？协议怎么设计？

来源：2019、2020、2022。

题目：

> 多个传感器从不同位置监控系统温度，最终温度取正确传感器测得值的平均。故障传感器发送任意值，无法仅凭收到值判断是否故障。设计协议保证正确传感器协调一致，并说明 K 个故障时需要多少传感器。

答案：

```text
协议：
1. 每个传感器向所有其他传感器广播自己测得的温度值。
2. 每个传感器收集所有传感器的值，形成包含自身在内的值向量。
3. 每个传感器再向所有其他传感器广播该向量。
4. 对每个传感器 i，检查收到的所有向量第 i 个元素；如果某个值占多数，则认为 i 的值为该多数值。
5. 将能确定的正确传感器值取平均，得到最终温度。

为了容忍 K 个拜占庭式故障传感器，常见要求总数至少 3K+1。
原因是故障节点可能向不同正确节点发送不同值，需要足够多正确节点通过多轮交换识别一致信息并达成相同结论。
```

易错点：

- 这里不是唯一正确结果的多数投票，而是要识别一组正确传感器值并求平均，故障节点可能说谎，常见要求 3K+1。

### 39. 25 个节点可靠性分析：最多能容忍多少错误节点？

来源：2022。

题目：

> 25 个节点，错误节点会返回随机值。1）正确节点返回的正确结果相同，允许最大容错节点数是多少？2）正确节点返回的正确结果不同，需要收集正确节点结果再取平均，允许最大容错节点数是多少？

答案：

```text
1）唯一正确结果相同：
使用 2K+1 <= 25。
K <= 12。
因此最多容忍 12 个错误节点。

2）正确结果不同，需要识别正确节点集合并取平均：
按拜占庭协同常见要求 3K+1 <= 25。
K <= 8。
因此最多容忍 8 个错误节点。
```

易错点：

- 第一问是多数投票；第二问是拜占庭式协同/平均值问题。

### 40. 2PC 正常流程、超时和协调者宕机如何处理？

来源：2022；最新 PDF 主线。

题目：

> 使用 2PC 进行分布式事务提交。1）所有参与者 OK 后谁发送什么？2）若未按时收到某参与者 OK，接下来发生什么？3）协调者已发送 phase2-commit 后宕机，重启看到的事务状态应怎样？

答案：

```text
2PC 分为投票阶段和决定阶段。

1）当协调者收到所有参与者 OK 后，进入第二阶段，协调者向所有参与者发送 phase2-commit / commit 消息。

2）若协调者在第一阶段未按时收到某参与者 OK，说明不是所有参与者都确认可提交。
为了保证原子性，协调者应决定 abort，并向已响应或可联系的参与者发送 abort。

3）如果协调者已经写入 commit 决定并发送 phase2-commit 后宕机，
重启后应根据本地稳定日志恢复出事务状态为 commit，并继续向参与者重发 commit，直到确认完成。
原因是 2PC 的决定必须持久化，否则协调者恢复后无法保证所有参与者结果一致。
```

易错点：

- 协调者发 commit 前必须先把 commit 决定写入稳定日志。
- 参与者投 yes 后若没收到最终决定会阻塞，不能随便 abort。

### 41. 为什么 2PC 是阻塞性的？

来源：Peanuts 补充；2PC 概念边界题。

题目：

> 为什么说 two-phase commit 是阻塞协议？阻塞发生在什么状态？

答案：

```text
2PC 的阻塞主要发生在参与者投 yes 并进入 prepared 状态之后。
此时参与者已经承诺可以提交，通常需要保留锁和日志，不能单方面 abort；
但如果协调者在发送最终 commit/abort 前崩溃，参与者又不知道全局决定，就只能等待协调者恢复或向其他参与者询问。

因此，2PC 能保证原子提交，但在协调者故障和网络不确定时可能导致参与者阻塞，占用资源。
```

易错点：

- 阻塞不是指“两阶段所以慢”，而是指 prepared 参与者不知道最终决定时不能自行结束。
- 超时可以帮助检测异常，但不能在所有情况下安全消除阻塞。

### 42. 3PC 相比 2PC 解决什么问题？

来源：最新 PDF 主线。

题目：

> Three-phase commit 的基本阶段是什么？它相对 2PC 想解决什么问题？

答案：

```text
3PC 通常包括 CanCommit、PreCommit 和 DoCommit 三个阶段。
CanCommit 阶段询问参与者是否可能提交；
PreCommit 阶段在所有参与者同意后，让参与者进入预提交状态；
DoCommit 阶段发送最终提交决定。

3PC 的目标是在 2PC 基础上减少协调者故障导致的阻塞。
但 3PC 依赖更强的超时和网络假设，实现更复杂，并不是在所有异步故障场景下都绝对无阻塞。
```

易错点：

- 不要写“3PC 永不阻塞”。应写“减少阻塞风险，需要更强假设”。

### 43. 2PC 和 3PC 的概念、优劣如何对比？

来源：Peanuts 补充。

题目：

> 比较 2PC 和 3PC 的基本阶段、优点、缺点，并说明 3PC 是否完全解决阻塞。

答案：

```text
2PC 分为投票阶段和决定阶段。
协调者先询问所有参与者是否可以提交；所有人都 yes 则广播 commit，否则广播 abort。
2PC 保证原子提交，协议简单、实现成熟；
缺点是参与者投 yes 后进入 prepared 状态，如果协调者故障且参与者未收到最终决定，就会阻塞。

3PC 在 2PC 基础上增加 pre-commit 阶段，通常分为 CanCommit、PreCommit 和 DoCommit。
它试图让参与者在最终提交前知道全局已经准备提交，从而在某些协调者故障场景下降低阻塞风险。
但 3PC 依赖更强的网络和超时假设，不能在完全异步网络和任意分区下彻底消除阻塞，复杂度也更高。
```

易错点：

- 3PC 是“降低阻塞风险”，不是无条件解决所有阻塞。
- 实际系统中 2PC 常配合日志、超时、恢复和协调者高可用使用。

### 44. 回退恢复、前向恢复和 checkpoint

来源：最新 PDF 主线；2020 出过 checkpoint 协议。

题目：

> 什么是回退恢复、前向恢复和 checkpoint？如何利用 checkpoint 做恢复？

答案：

```text
回退恢复是在故障后把系统恢复到之前保存的正确状态，再从该状态重新执行。
前向恢复是不回到过去状态，而是通过补偿、修复或状态转换把系统推进到新的正确状态。
Checkpoint 是运行过程中保存的一致状态快照，用作恢复点。

利用 checkpoint 的恢复流程：
1. 系统定期保存一致 checkpoint。
2. 发生故障后检测错误状态。
3. 选择最近的全局一致 checkpoint。
4. 各进程回滚到该 checkpoint，并重放必要日志或重新执行。
5. 从恢复点继续运行。
```

易错点：

- checkpoint 要是一致状态，否则恢复后可能出现消息丢失或重复等问题。

## 六、Paxos、Raft 与应用补充

### 45. Paxos 的假设、活锁和解决方法

来源：2021、2020 讲座题；往年补充。

题目：

> Paxos 协议中，Basic Paxos 的基本假设是什么？什么情况下会活锁？如何解决？

答案：

```text
Basic Paxos 假设节点可能故障、消息可能延迟或丢失，但非拜占庭故障；多数派节点可通信时可达成一致。
它通过 proposal number 和多数派 quorum 保证不会选出两个不同值。

活锁场景：
多个 proposer 同时提出更高编号提案，彼此不断抢占。
一个 proposer 刚获得部分 promise，另一个 proposer 用更高编号使其提案失效。
系统一直有消息活动，但迟迟不能选定值。

解决方法：
选举或指定一个稳定 leader，由 leader 统一提出提案，减少 proposer 竞争。
```

易错点：

- 活锁不是死锁；系统在运行但没有有效进展。
- Basic Paxos 不处理拜占庭故障；活锁通常来自多个 proposer 竞争和不断抢占。
- 最新 PDF 未显式列 Paxos，若时间紧可低于 2PC/3PC。

### 46. Raft 三部分过程

来源：2021、2022；往年补充。

题目：

> Raft 可分为领导人选举、日志复制、安全性三个部分，请简述每个部分过程。

答案：

```text
1. 领导人选举：
节点初始为 follower。若选举超时未收到 leader 心跳，转为 candidate，增加 term 并请求投票。
同一 term 每个节点最多投一票，候选者获得多数票后成为 leader。
Leader 定期发送心跳维持权威。

2. 日志复制：
客户端请求由 leader 处理。
Leader 将命令追加到本地日志，通过 AppendEntries RPC 复制给 follower。
日志复制到多数派后，leader 提交该日志并应用到状态机，再通知 follower 提交。

3. 安全性：
Raft 依赖 term、日志匹配和多数派交集。
已提交日志存储在多数派上，新 leader 也必须获得多数派支持，因此会与旧多数派相交。
选举限制要求候选者日志足够新，避免缺少已提交日志的节点成为 leader。
```

易错点：

- “多数派”是选举和提交的共同核心。
- 最新 PDF 未显式列 Raft，但 2022/2021 出过，建议保留补分。

### 48. 非交叉传输和交叉传输的优缺点

来源：2020。

题目：

> 分别描述流媒体非交叉传输和交叉传输的优缺点。

答案：

```text
非交叉传输按媒体播放顺序连续发送数据。
优点是实现简单、处理开销小、延迟低；
缺点是突发丢包会影响连续片段，抗丢包能力较弱。

交叉传输将相邻播放数据分散交错传输。
优点是突发丢包造成的损失被分散，抗丢包能力更强；
缺点是接收端需要重排和缓冲，实现复杂，可能增加启动延迟。
```

易错点：

- 交叉传输提升抗突发丢包能力，但不是免费，会增加缓冲和重排成本。

### 49. 如何提升在线直播流畅性？

### 51. 并行策略和并行编程范式有哪些？

```text
常见并行策略包括：
1. 数据并行：把数据切成多个分片，不同节点对不同分片执行相同操作，适合批处理、训练和统计计算。
```

### 52. 大模型并行训练方法及通信特点

来源：2022/2024 讲座题；往年补充。

```text
并行训练方法：
1. 数据并行：每个节点保存完整模型，处理不同数据分片，同步梯度。
```

### 53. 虚拟化解决什么问题？与云计算是什么关系？

来源：Peanuts 补充；云计算基础题。

题目：

```text
虚拟化把物理 CPU、内存、存储和网络抽象成虚拟资源，解决资源利用率低、部署迁移复杂、隔离困难和弹性扩展困难等问题。
多个虚拟机或容器可共享同一物理资源，同时保持运行环境隔离。
虚拟化支持快速创建、迁移、复制和回收实例，因此是云计算资源池化、按需分配和弹性伸缩的基础。
云计算则在虚拟化基础上，通过自动化管理向用户提供 IaaS、PaaS、SaaS 等服务。
```

易错点：

- 虚拟化是云计算的重要基础，但云计算还包括资源调度、计费、自动化运维、服务接口等管理能力。
- 容器也是一种轻量级虚拟化/隔离技术，但它不等同于完整虚拟机。

### 54. 数据中心为什么需要容灾？有哪些方式？

来源：2022。

题目：

> 数据中心为何要建立容灾机制？容灾可以采用哪些方式？

答案：

```text
数据中心需要容灾，因为单个机房可能发生硬件故障、网络故障、电力/制冷故障、自然灾害、软件缺陷或人为误操作。
容灾目标是在故障发生时尽量保证服务可用、数据不丢失或少丢失，并缩短恢复时间。

常见方式包括：
1. 定期备份与恢复。
2. 主备部署和故障切换。
3. 同城双活。
4. 异地多活。
5. 跨机房数据复制。
6. 灾难恢复演练和监控告警。
```

易错点：

- 容灾不是只备份数据，还要考虑服务切换、恢复时间和数据一致性。

## 七、最后冲刺建议

如果只剩很短时间，按下面顺序刷：

1. 定义/目标/透明性/开放性/OS vs 中间件。
2. TCP/UDP、同步/异步、持久/瞬时、RPC 故障、at-most-once。
3. Lamport/向量时钟规则和判断。
4. 一致性模型、门户网站、Quorum。
5. `2K+1` / `3K+1`、2PC/3PC、恢复/checkpoint。
6. 流媒体 QoS、Raft、Paxos、大模型、容灾作为补分。
