# 模块六：应用与讲座题

> 时间紧时把本模块当补分题准备：流媒体 QoS、交叉/非交叉传输、虚拟化与云计算、Hadoop/Spark/YARN、大模型训练、数据中心容灾。最新 `Review-Distributed System.pdf` 只显式提到通信里的 stream data，没有把 Spark/YARN、RFID、大模型列为主线；这些内容主要按往年题和旧资料保留。

## 1. 本模块主线

应用题通常不要求推协议细节，更看重：

```text
背景是什么 -> 分布式挑战是什么 -> 常见方案是什么 -> 为什么有效。
```

答题时优先写关键词，不要展开到工程论文级细节。

## 2. 流媒体 QoS

QoS 指标必须会默写：

1. 数据传输所需比特率。
2. 会话创建最大延迟。
3. 端到端最大延迟。
4. 最大延迟抖动。
5. 最大往返延迟。

标准答案：

```text
流媒体 QoS 关注用户播放体验，常见指标包括数据传输所需比特率、会话创建最大延迟、端到端最大延迟、最大延迟抖动和最大往返延迟。
比特率影响画质，端到端延迟和往返延迟影响实时性，延迟抖动会导致播放卡顿，因此需要缓存、调度和传输优化。
```

## 3. 如何提升在线直播流畅性

可写方法：

- 使用 CDN 或边缘节点，就近分发内容。
- 使用 P2P 辅助分发，减轻中心服务器压力。
- 使用播放缓冲区，吸收网络抖动。
- 使用自适应码率，根据网络状况切换清晰度。
- 使用交叉传输或冗余/FEC，提高抗丢包能力。
- 优化调度，将用户分配到延迟更低、负载更低的节点。

标准答案：

```text
在不改变物理网络环境的情况下，可通过 CDN 或边缘节点将直播流就近分发，降低访问延迟并减轻源站压力。
也可使用 P2P 辅助分发，让用户之间互相转发部分数据，提高带宽利用率。
客户端可设置播放缓冲区吸收网络抖动，并使用自适应码率根据网络状况降低或提升清晰度。
对于丢包场景，可采用交叉传输、冗余包或前向纠错，提高连续播放能力。
```

## 4. 非交叉传输 vs 交叉传输

| 类型 | 含义 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 非交叉传输 | 按媒体播放顺序连续传输 | 实现简单、延迟低、开销小 | 连续丢包时影响一段连续播放 |
| 交叉传输 | 将相邻播放数据分散交错传输 | 抗突发丢包能力更强 | 需要重排和缓冲，实现复杂，可能增加延迟 |

标准答案：

```text
非交叉传输按媒体播放顺序连续发送数据，实现简单、处理开销小、延迟较低，但遇到突发丢包时可能导致连续片段无法播放。
交叉传输将相邻播放数据分散到不同传输位置，突发丢包造成的损失会被分散，抗丢包能力更强。
但交叉传输需要接收端重排和更多缓冲，实现复杂，并可能增加启动延迟。
```

## 5. 虚拟化与云计算

### 5.1 为什么云计算依赖虚拟化

标准答案：

```text
虚拟化将物理计算、存储和网络资源抽象成可按需分配的虚拟资源。
云计算依赖虚拟化实现资源池化、隔离、弹性伸缩和按需使用。
通过虚拟机或容器，多个用户和应用可以共享同一批物理资源，同时保持运行环境隔离，并可快速创建、迁移和回收实例。
```

### 5.2 私有云

```text
私有云是由单个组织独占使用的云基础设施，可部署在组织内部或由第三方托管。
它保留云计算的资源池化、弹性和自动化能力，同时更便于满足安全、合规和内部管理要求。
```

## 6. Hadoop、Spark、YARN

### 6.1 Hadoop

标准答案：

```text
Hadoop 是面向大规模数据存储和批处理的分布式系统。
HDFS 负责把大文件分块并复制到多台机器上，提高存储容量和容错能力；
MapReduce 负责把计算任务拆分到数据所在节点附近执行，再汇总结果。
它适合大规模离线批处理，但迭代计算和交互式计算效率较低。
```

### 6.2 Spark

标准答案：

```text
Spark 是面向大规模数据处理的分布式计算框架，核心特点是基于内存计算和 DAG 执行模型。
相比 MapReduce 每轮都频繁落盘，Spark 可把中间结果缓存在内存中，因此更适合迭代计算、机器学习和交互式分析。
DAG 调度器能根据任务依赖关系优化执行计划，提高作业执行效率。
```

### 6.3 YARN

标准答案：

```text
YARN 是 Hadoop 生态中的资源管理和任务调度系统。
它将资源管理和计算框架解耦，由 ResourceManager 统一管理集群资源，NodeManager 管理单节点资源，ApplicationMaster 负责单个应用的调度协调。
这样 MapReduce、Spark 等不同计算框架可以共享同一集群资源。
```

## 7. YARN pull-based vs Mesos push-based

常见对比：

| 项目 | YARN pull-based | Mesos push-based |
| --- | --- | --- |
| 资源获取 | 应用向 ResourceManager 请求资源 | Mesos 向框架提供 resource offer |
| 决策方式 | 应用提出需求，资源管理器分配 | 框架决定是否接受资源 offer |
| 特点 | 适合 Hadoop 生态，模型清晰 | 多框架共享资源更灵活 |

可背答案：

```text
YARN 更偏 pull-based，应用的 ApplicationMaster 根据任务需求向 ResourceManager 申请资源，ResourceManager 再分配容器。
Mesos 更偏 push-based，资源管理器把可用资源以 offer 形式推给框架，框架根据自身需求决定是否接受。
两者都解决集群资源共享问题，但资源分配方向和调度决策位置不同。
```

## 8. Spark + YARN 对 DAG 作业的优势

标准答案：

```text
Spark 使用 DAG 描述作业中各阶段的依赖关系，可根据依赖关系进行任务划分、流水执行和失败重算。
YARN 提供统一资源管理，使 Spark 可以和其他计算框架共享集群资源。
Spark + YARN 结合后，既能利用 YARN 的资源调度和隔离能力，又能利用 Spark 的内存计算和 DAG 优化能力，
因此适合大规模迭代计算和复杂数据分析作业。
```

## 9. RFID / Aloha

### 9.1 Aloha 协议

标准答案：

```text
Aloha 是一种随机接入协议，节点有数据时直接发送。
如果发生冲突，节点等待随机时间后重传。
其优点是简单、分布式、无需集中调度；缺点是负载较高时冲突增多，信道利用率下降。
```

### 9.2 RFID 阅读器选择问题

```text
RFID 系统中多个阅读器可能覆盖同一标签，若同时读取会产生干扰或重复读取。
阅读器选择问题就是在保证覆盖目标标签的前提下，选择合适阅读器集合或调度顺序，减少冲突、冗余和能耗。
集中式方案全局优化能力强但依赖中心节点；分布式方案扩展性和鲁棒性更好，但可能难以获得全局最优。
```

## 10. 大模型并行训练

### 10.1 为什么是分布式系统问题

```text
大模型训练需要处理海量参数和数据，单机通常无法承载全部计算、显存和存储需求。
因此需要把数据、模型参数或计算过程分布到多台 GPU/机器上，并通过高速网络进行同步。
这涉及通信、同步、资源调度、容错和存储等典型分布式系统问题。
```

### 10.2 并行训练方法

常见方法：

| 方法 | 含义 |
| --- | --- |
| 数据并行 | 每个节点保存完整模型，处理不同数据分片，同步梯度 |
| 模型并行 | 将模型参数或层切分到不同设备 |
| 流水线并行 | 将模型层划分为阶段，不同 mini-batch 流水执行 |
| 张量并行 | 将单层内部矩阵计算切分到多个设备 |
| 混合并行 | 组合数据、模型、流水线、张量并行 |

标准答案：

```text
大模型并行训练常见方法包括数据并行、模型并行、流水线并行、张量并行和混合并行。
数据并行让多个节点处理不同数据分片并同步梯度；
模型并行把模型不同部分放到不同设备；
流水线并行把模型层划分为多个阶段并流水处理不同批次；
张量并行进一步切分单层内部矩阵计算。
实际训练通常组合多种并行方式以适应显存、计算和通信限制。
```

### 10.3 大模型训练通信的新特点

标准答案：

```text
大模型训练通信相比普通分布式业务系统更密集、更规律、带宽需求更高。
训练过程中需要频繁同步梯度、参数或激活值，通信量巨大，且常位于计算关键路径上。
普通业务系统通信多是请求响应或消息传递，更关注延迟、可用性和事务语义；
大模型训练更关注高带宽、低延迟、集合通信效率、通信与计算重叠以及故障恢复。
```

挑战可写：

- 通信瓶颈。
- GPU/网络资源调度。
- 节点故障恢复。
- Checkpoint 存储压力。
- 负载均衡和同步等待。

## 10.4 并行策略与并行编程范式

讲座题如果问“并行策略以及并行编程范式”，不要只写大模型训练。可以从更通用的并行计算角度回答。

常见并行策略：

| 策略 | 含义 | 例子 |
| --- | --- | --- |
| 数据并行 | 同一操作作用于不同数据分片 | MapReduce、批量图片处理 |
| 任务并行 | 不同任务或函数并发执行 | Web 服务不同请求并发处理 |
| 流水线并行 | 把处理过程拆成阶段，不同数据批次在不同阶段同时运行 | 视频处理流水线、模型流水线 |
| 分治并行 | 把大问题拆成子问题并行求解再合并 | 并行排序、矩阵计算 |
| 参数/模型并行 | 把模型或状态切分到多个节点 | 大模型训练 |

并行编程范式：

| 范式 | 核心思想 | 适合场景 |
| --- | --- | --- |
| Message passing | 进程通过显式消息通信 | MPI、分布式内存集群 |
| Shared memory | 多线程共享地址空间，通过锁/同步协调 | 单机多核并行 |
| Dataflow / DAG | 用依赖图描述计算，系统按依赖调度 | Spark、TensorFlow |
| MapReduce | map 处理分片，reduce 汇总结果 | 离线批处理 |
| Actor | 每个 actor 通过消息维护私有状态 | 高并发服务、事件系统 |

答题模板：

```text
并行策略描述如何把计算拆开，常见有数据并行、任务并行、流水线并行、分治并行和模型/参数并行。
数据并行把不同数据分片交给多个节点执行相同操作；任务并行让不同任务同时运行；
流水线并行把处理流程拆成阶段；分治并行先拆子问题再合并结果；模型并行把大模型或大状态切到不同设备。
并行编程范式描述程序如何表达并行，常见包括消息传递、共享内存、Dataflow/DAG、MapReduce 和 Actor。
答题时可结合场景说明：分布式内存集群常用消息传递，数据分析常用 MapReduce 或 DAG，大规模训练常组合数据并行、模型并行和流水线并行。
```

## 11. 数据中心容灾

为什么需要：

- 硬件故障。
- 网络故障。
- 电力/制冷故障。
- 机房级灾害。
- 人为误操作。
- 软件缺陷或配置错误。

常见方式：

| 方式 | 含义 |
| --- | --- |
| 备份恢复 | 定期备份数据，故障后恢复 |
| 主备 | 主中心服务，备中心待命或同步 |
| 同城双活 | 同城两个中心同时服务或快速切换 |
| 异地多活 | 多地域同时承载流量 |
| 数据复制 | 同步或异步复制关键数据 |
| 故障切换 | 故障时自动或手动切到备用系统 |

标准答案：

```text
数据中心需要容灾，因为单个机房可能发生硬件、网络、电力、自然灾害、软件缺陷或人为误操作等故障。
容灾目标是在故障发生时尽量保证服务可用、数据不丢失或少丢失，并缩短恢复时间。
常见方式包括数据备份、主备部署、同城双活、异地多活、跨机房数据复制和故障切换。
不同方案在成本、复杂度、恢复时间和数据一致性之间存在权衡。
```

## 12. 常见答题模板

### 12.1 应用题通用模板

```text
该场景的核心问题是多节点环境下的资源共享、通信和一致性/可用性权衡。
主要挑战包括网络延迟、节点故障、负载不均、数据同步和资源调度。
常见方案是通过复制、缓存、调度、分片、容错和监控恢复机制提升性能和可靠性。
代价是系统复杂度增加，并需要在一致性、可用性、成本和性能之间做取舍。
```

### 12.2 大模型训练题模板

```text
大模型训练需要把数据、模型和计算分布到多台 GPU/机器上，原因是单机显存、算力和存储不足。
常见并行方式包括数据并行、模型并行、流水线并行、张量并行和混合并行。
相比普通分布式业务通信，大模型训练通信更频繁、带宽需求更高，常涉及 AllReduce 等集合通信，并位于训练关键路径。
因此主要挑战是通信瓶颈、同步等待、资源调度、故障恢复和 checkpoint 存储压力。
```
