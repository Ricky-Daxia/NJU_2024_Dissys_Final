# 模块二：通信与 RPC

> 时间紧时只背这一版：通信类型、TCP/UDP、同步/异步、持久/瞬时、RPC 工作过程与故障、动态绑定、基于消息通信和流数据。最新 PDF 明确把“基于消息的通信”和“流数据”列入范围。

## 1. 本模块主线

RPC 看起来像本地函数调用，但本质是跨机器通信，所以会出现本地调用没有的不确定性：

- 请求可能丢失。
- 响应可能丢失。
- 服务器可能执行到一半崩溃。
- 客户端可能崩溃，服务器还在执行。
- 客户端超时后不知道请求到底有没有被执行。

核心句：

```text
RPC 超时不等于失败，重试也不一定安全。
是否能重试取决于操作是否幂等、服务端是否去重，以及调用语义是 at-least-once 还是 at-most-once。
```

## 2. TCP 与 UDP

位置：

> TCP 和 UDP 都位于传输层。

对比：

| 项目 | TCP | UDP |
| --- | --- | --- |
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠、有序、无重复的字节流 | 不保证可靠，不保证顺序 |
| 控制 | 有流量控制和拥塞控制 | 控制少，开销低 |
| 适用 | 可靠 RPC、HTTP/gRPC、文件传输 | 实时音视频、DNS、应用层自定义可靠性 |

标准答案：

```text
TCP 和 UDP 都位于传输层。
TCP 是面向连接的可靠字节流协议，提供按序、可靠传输，并带有流量控制和拥塞控制，适合可靠性要求高的 RPC。
UDP 是无连接协议，开销小、延迟低，但不保证可靠性和顺序，适合实时性要求高、允许丢包或由应用层自行处理可靠性的场景。
```

易错点：

```text
不要说“TCP 是同步通信”。
TCP 是传输层协议；同步/异步描述的是应用层调用方式。
阻塞式 socket 或同步 RPC 可能表现为等待响应，但这不是 TCP 本身的语义。
```

## 3. 同步/异步，持久/瞬时

### 3.1 同步通信与异步通信

```text
同步通信中，发送方发送请求后等待接收方响应，才能继续后续逻辑。
异步通信中，发送方发送请求后可以继续执行，不必立即等待响应，结果可通过回调、通知、轮询或消息队列返回。
二者区别在应用层控制流是否阻塞，不由 TCP 或 UDP 直接决定。
```

后端例子：

- 同步：普通 HTTP 请求、gRPC unary 调用。
- 异步：发送 MQ 消息、异步任务、Future/Promise。

### 3.2 持久通信与瞬时通信

```text
瞬时通信要求发送和接收进程在通信期间同时运行，通信系统不会长期保存消息。
持久通信中，消息提交后可由通信中间件保存，即使接收方暂时未运行，消息也可之后投递。
普通 RPC 多属于瞬时通信，消息队列和邮件系统是持久通信的典型例子。
```

例子：

```text
电子邮件属于持久通信，因为邮件提交给邮件服务器后，即使接收方不在线，邮件也会被服务器保存。
QQ 留言也可按持久通信理解，因为留言会被服务器保存，接收方之后上线再读取。
普通在线聊天如果只在双方在线时投递、离线即丢弃，则更接近瞬时通信。
```

## 4. RPC 基本过程

最小流程：

1. 客户端调用 client stub。
2. client stub 将参数编组为请求消息。
3. 请求通过网络发给服务器。
4. server stub 解组参数并调用真实服务过程。
5. 服务执行后返回结果。
6. 响应返回客户端，client stub 解组得到返回值。

标准答案：

```text
RPC 通过客户端存根和服务器存根隐藏网络通信细节。
客户端调用 client stub，stub 将参数编组为消息并发送给服务器；
服务器端 stub 解组消息并调用实际过程；执行结果再被编组为响应返回客户端。
因此远程过程调用在形式上类似本地过程调用，但故障语义不同。
```

## 5. RPC 五类常见故障

必须会默写：

1. 客户端无法定位服务器。
2. 客户端到服务器的请求消息丢失。
3. 服务器到客户端的响应消息丢失。
4. 服务器接收请求后发生故障。
5. 客户端发送请求后发生故障。

标准答案：

```text
RPC 的故障包括：客户端无法定位服务器；请求消息丢失；响应消息丢失；
服务器收到请求后在执行前、执行中或执行后发生故障；客户端发送请求后崩溃，导致服务器端可能出现孤儿计算。
这些故障使客户端很难仅凭超时判断请求到底是未到达、已执行但响应丢失，还是执行过程中失败。
```

## 6. 请求丢失与响应丢失

### 6.1 请求消息丢失

处理思路：

- 客户端设置超时。
- 超时后可重发。
- 重发要配合请求 ID、幂等或服务端去重。

标准答案：

```text
请求消息丢失时，客户端通常无法直接知道请求是否到达服务器。
常见方法是设置超时并重发请求。
为了避免重复执行带来副作用，请求应带唯一请求 ID，服务器可记录已处理请求并去重，或者要求操作本身是幂等的。
```

### 6.2 响应消息丢失

关键风险：

> 服务器可能已经执行成功，只是响应丢了。客户端超时后重试，可能重复执行。

标准答案：

```text
响应消息丢失比请求丢失更麻烦，因为服务器可能已经执行了操作。
如果客户端超时后简单重试，可能导致操作重复执行。
常见解决方法是为每个请求设置唯一编号，服务器记录已处理请求及其结果；
当重复请求到达时，服务器不再次执行操作，而是返回缓存结果或确认该请求已经处理。
```

后端例子：

- 查询订单可以重试。
- 创建订单、扣款、发券不能盲目重试，必须有幂等键或业务流水号。

## 7. 服务器接收请求后故障

客户端无法判断服务器处于哪种状态：

- 请求还没执行，服务器就崩溃。
- 请求执行到一半崩溃。
- 请求执行完成，但响应还没返回就崩溃。
- 请求已产生副作用，但客户端不知道。

标准答案：

```text
服务器接收请求后故障会导致客户端无法判断请求是否已经执行以及执行结果是否持久化。
如果采用 at-least-once 语义，客户端可超时重试，但可能导致重复执行。
如果采用 at-most-once 语义，需要请求编号、服务端日志或结果缓存来避免重复执行。
对于有副作用的操作，还需要事务、幂等设计或恢复日志来保证崩溃恢复后的状态一致。
```

后端类比：

- 支付请求超时后不能简单再发一次。
- 更稳妥做法是业务流水号 + 幂等表 + 状态查询/补偿。

## 8. 客户端发送请求后故障

问题名称：

> orphan computation，孤儿计算。

含义：

```text
客户端崩溃或放弃请求后，服务器仍在执行该客户端发起的远程计算。
结果可能无人接收，还可能继续占用资源或产生副作用。
```

四种处理方法：

| 方法 | 含义 |
| --- | --- |
| Extermination | 客户端调用前写日志，恢复后根据日志清理孤儿 |
| Reincarnation | 客户端重启后广播新 epoch，杀死旧 epoch 的远程计算 |
| Gentle reincarnation | 先检查远程计算是否仍有 owner，无 owner 再清理 |
| Expiration | 为 RPC 设置租约/超时时间，到期未续约则终止 |

标准答案：

```text
客户端发送请求后崩溃可能导致服务器继续执行孤儿计算。
处理方法包括：调用前记录日志，客户端恢复后根据日志消灭孤儿；
客户端重启后广播新的时期号，要求清理旧时期的远程计算；
采用温和再生方式，先检查计算是否仍有拥有者，若无拥有者再终止；
或者为 RPC 设置租约，到期未续约则自动结束。
```

## 9. maybe、at-least-once、at-most-once

对比：

| 语义 | 含义 | 风险 |
| --- | --- | --- |
| maybe | 最多尝试一次，失败不重试 | 可能根本没执行 |
| at-least-once | 超时重试，尽量至少执行一次 | 可能重复执行 |
| at-most-once | 请求 ID + 去重，最多执行一次 | 可能没有执行成功 |

标准答案：

```text
maybe 语义表示客户端发送一次请求，若失败不保证是否执行。
at-least-once 语义通过超时重试尽量保证请求至少执行一次，但可能因为响应丢失或重试导致重复执行。
at-most-once 语义通过唯一请求 ID、服务端去重和结果缓存，保证同一请求不会被重复执行，但不保证请求一定成功执行。
```

at-most-once 常见实现：

1. 客户端为每次请求生成唯一请求 ID。
2. 服务端维护已处理请求表。
3. 第一次收到请求时执行，并记录结果。
4. 重复请求到达时不重新执行，直接返回缓存结果或“已处理”。

易错点：

- at-most-once 不是“一定执行一次”，而是“最多执行一次”。
- at-least-once 不是“不会重复”，而是“可能重复”。
- exactly-once 通常需要限定条件，工程里多靠幂等、事务、去重和确认机制组合实现。

## 10. 动态绑定与服务器定位

为什么需要：

> 如果把服务器地址写死在客户端，服务器迁移、扩容、升级都会很困难。

标准答案：

```text
动态绑定用于解决服务器定位问题。
客户端不把服务器地址固定写入程序，而是在调用前通过 binder 或注册中心查询服务实例位置。
这样可以支持服务器迁移、扩容、多版本和故障切换。
缺点是增加查询开销，并且 binder 或注册中心本身需要避免成为单点故障。
```

后端类比：

- Nacos、Consul、etcd、ZooKeeper、Kubernetes Service 都类似服务发现机制。

答题模板：

```text
动态绑定用于在运行时把客户端请求绑定到合适的服务器实例，而不是在程序中写死服务器地址。
服务端启动时向 binder 或注册中心导出接口和地址，客户端调用前先向 binder 查询可用服务实例，再发起 RPC。
这样可以支持服务器迁移、扩容、负载均衡、容错切换和接口版本检查。
缺点是多了一次查询和维护开销，binder 本身可能成为瓶颈或单点故障，因此也需要复制、缓存或高可用设计。
```

易错点：

- 动态绑定解决“运行时找到谁调用”的问题，不等于自动解决 RPC 的执行语义。
- binder 能帮助定位和版本匹配，但不保证被调用服务一定成功执行。

## 11. RPC 为什么不能等同本地调用

这题很适合作为大题总结答案：

```text
RPC 在形式上隐藏了网络通信，使远程调用看起来像本地调用，但二者语义不同。
本地调用的失败通常较明确，而 RPC 可能遇到服务器定位失败、请求丢失、响应丢失、服务器崩溃和客户端崩溃等问题。
客户端超时后无法判断请求是否已执行，因此重试可能导致重复副作用。
所以 RPC 系统必须考虑超时、重试、请求编号、幂等、去重、故障恢复和孤儿计算清理等问题。
```

## 12. 基于消息的通信

一句话：

> 基于消息的通信通过显式发送和接收消息来交换数据，通信双方不必像 RPC 那样表现为一次函数调用。

对比：

| 项目 | RPC | 消息通信 |
| --- | --- | --- |
| 抽象 | 像调用远程函数 | 显式发送/接收消息 |
| 耦合 | 调用方通常等待服务方语义 | 可以更松耦合 |
| 典型形式 | request/reply | message queue、publish/subscribe |
| 适用 | 同步业务调用 | 异步任务、事件通知、削峰填谷 |

标准答案：

```text
基于消息的通信中，进程通过发送和接收消息进行交互。
与 RPC 将远程通信包装成本地调用不同，消息通信显式暴露消息传递过程，可支持队列、发布订阅和异步处理。
如果通信系统保存消息直到接收者可用，则属于持久通信；如果只在发送者和接收者同时运行时传递，则属于瞬时通信。
```

后端例子：

- MQ 下单后异步扣库存或发送通知。
- 发布订阅用于配置变更、事件广播。

易错点：

- 消息通信不等于一定异步，也可以做 request-reply。
- MQ 的 at-least-once 投递仍可能重复消费，需要幂等。

### 12.1 消息队列和消息代理

PPT 常见操作：

| 操作 | 含义 |
| --- | --- |
| PUT | 将消息放入队列 |
| GET | 阻塞等待并取出消息 |
| POLL | 非阻塞检查是否有消息 |
| NOTIFY | 消息到达后通知接收方 |

```text
消息代理 message broker 是位于通信双方之间的中间件组件。
它可以保存、路由、过滤和转换消息，使使用不同数据格式或协议的应用仍能协同工作。
消息代理的好处是解耦发送方和接收方，缺点是 broker 本身需要处理性能瓶颈、可靠性和单点故障问题。
```

知识点展开：

```text
消息队列把发送和接收解耦。发送方只需把消息放入队列，不要求接收方此刻正在运行；
接收方之后再从队列中获取消息。因此消息队列通常属于持久异步通信。
消息代理比普通队列更进一步，它可以根据主题、路由规则或消息内容把消息分发给不同接收者，
也可以在异构系统之间做格式转换，例如把一种应用的消息格式转换成另一种应用能理解的格式。
```

消息代理常见能力：

| 能力 | 含义 | 考试可写例子 |
| --- | --- | --- |
| Routing | 根据主题、队列或规则转发消息 | 订单事件发给库存和通知服务 |
| Transformation | 转换消息格式或协议 | XML 转 JSON，不同字段名映射 |
| Buffering | 接收方不可用时暂存消息 | 邮件系统、离线任务队列 |
| Filtering | 只把符合条件的消息发给订阅者 | 只订阅某类告警 |

答题模板：

```text
消息队列通过 PUT、GET、POLL、NOTIFY 等操作支持进程之间的持久消息通信。
发送方把消息放入队列后即可继续执行，接收方可在之后阻塞获取或轮询获取消息。
消息代理是更通用的通信中间件，它不仅保存消息，还能路由、过滤和转换消息，
使不同应用即使用不同格式或协议也能协同工作。
这种方式提升了解耦和可靠投递能力，但 broker 本身可能成为性能瓶颈或单点故障，需要复制和持久化机制支持。
```

易错点：

- 消息队列能缓解发送方和接收方同时在线的要求，但不自动保证 exactly-once。
- broker 可以降低应用之间耦合，但不会消除消息重复、乱序和积压问题。

## 13. 流数据与流通信

一句话：

> 流数据关注连续、按时间播放或处理的数据，核心问题是吞吐、延迟、抖动和连续性。

标准答案：

```text
流数据是随时间连续产生和消费的数据，例如音频、视频和实时监控数据。
流通信不只关心单条消息是否到达，还关心数据到达速率、端到端延迟、延迟抖动和播放连续性。
因此流媒体系统常用缓冲、自适应码率、丢包恢复和调度优化来保证用户体验。
```

与普通 RPC 的区别：

- RPC 更关注一次请求/响应是否成功。
- 流数据更关注持续速率、顺序、实时性和抖动。

PPT 补充：流同步、QoS 与应用层组播。

```text
流同步关注多个连续媒体流之间的时间关系，例如视频画面和音频声音要保持同步。
QoS 描述流通信需要满足的服务质量，包括带宽、端到端延迟、抖动、丢包率和同步误差。
系统可以通过缓冲、调度、资源预留、自适应码率和丢包恢复来尽量满足 QoS。
```

应用层组播常用两个指标：

| 指标 | 含义 |
| --- | --- |
| Link stress | 同一物理链路上被重复发送的相同数据份数 |
| Stretch | 应用层转发路径长度相对底层最短路径的膨胀程度 |

答题抓手：

```text
IP multicast 依赖网络层支持，部署困难；application-level multicast 在应用层组织覆盖网络，更容易部署，但可能造成链路重复传输和路径变长。
```

知识点展开：

```text
普通消息通信通常关心一条消息是否正确到达，而流通信关心一段连续数据在时间上的播放或处理质量。
例如视频会议中，晚到 5 秒的包即使最终到达也可能没有价值；音频和视频分别到达但不同步也会影响体验。
所以流数据需要关注带宽、延迟、抖动、丢包率、播放连续性和多流同步。
```

QoS 指标答题表：

| 指标 | 含义 | 影响 |
| --- | --- | --- |
| Bandwidth | 单位时间可传输的数据量 | 码率和清晰度 |
| Delay | 端到端传输耗时 | 实时交互体验 |
| Jitter | 延迟波动 | 播放卡顿和缓冲大小 |
| Loss rate | 丢包比例 | 画面/声音质量 |
| Synchronization skew | 多个流之间的时间偏差 | 音画同步 |

流同步标准答案：

```text
流同步是指保持多个连续数据流之间的时间关系。
例如视频会议中音频流和视频流可能走不同路径并有不同延迟，但播放时必须让声音和画面大致对齐。
系统通常通过时间戳、缓冲和播放调度来吸收网络抖动，并在延迟和连续性之间折中。
```

应用层组播答题模板：

```text
应用层组播不依赖路由器提供 IP multicast，而是在应用节点之间构造覆盖网络，由应用节点负责转发数据。
它部署更容易，适合互联网环境；但因为覆盖网络路径不一定等于底层最短路径，可能导致 stretch 变大；
同一物理链路上也可能重复传输相同数据，导致 link stress 变高。
```

易错点：

- QoS 不是只指“带宽高”，还包括延迟、抖动、丢包和同步。
- 流数据允许一定丢包，但通常不能容忍无限等待；实时性和可靠性要折中。

## 13.1 丢包场景下的可靠性改进

题目常会问“稳定 WiFi 下丢包”和“开放系统中的丢包”如何改进以满足可靠性保证。答题时要先区分：底层网络丢包只能通过通信协议和应用机制补偿，不能只说“换网络”。

| 机制 | 作用 | 适合场景 |
| --- | --- | --- |
| ACK/NAK | 确认收到或通知缺失 | 可靠文件、RPC、消息传输 |
| Timeout + retransmission | 超时未确认则重传 | 普通可靠传输 |
| Sequence number | 检测重复、乱序和丢失 | TCP、RPC 去重 |
| Sliding window | 在可靠性和吞吐之间折中 | 长连接、大文件传输 |
| Checksum | 检测比特错误 | 链路层/传输层 |
| FEC/冗余包 | 不等重传即可恢复部分丢包 | 实时音视频 |
| Buffer/interleaving | 吸收抖动，分散突发丢包影响 | 流媒体 |

答题模板：

```text
在稳定 WiFi 或开放网络中出现丢包时，可通过确认、超时重传、序列号、校验和滑动窗口实现可靠传输。
发送方为每个分组编号，接收方返回 ACK，发送方在超时未收到 ACK 时重传；
接收方利用序列号去重、重排并发现缺失分组。
如果是实时流媒体，完全依赖重传可能增加延迟，因此还可使用缓冲、交叉传输、冗余包或 FEC，在可靠性和实时性之间折中。
在开放系统中还要考虑异构网络和节点故障，可在应用层加入幂等、去重、持久队列和端到端确认。
```

易错点：

- TCP 已提供可靠字节流，但应用层 RPC 仍可能因超时重试导致重复执行，所以还要有请求 ID 和幂等。
- 实时音视频不一定追求每个包都重传成功，更重要的是连续播放和低延迟。

## 14. 考前最小背诵清单

- TCP/UDP 都在传输层。
- TCP：可靠、有序、面向连接；UDP：无连接、低开销、不保证可靠。
- 同步/异步是应用层调用方式，不是 TCP/UDP 本身语义。
- 持久通信会保存消息，瞬时通信不长期保存消息。
- RPC 工作过程：client stub 编组，请求发送，server stub 解组，执行后返回响应。
- RPC 五类故障：定位失败、请求丢失、响应丢失、服务器故障、客户端故障。
- 响应丢失最容易导致“服务器已执行但客户端重试”的重复副作用。
- at-least-once：超时重试，可能重复。
- at-most-once：请求 ID + 服务端去重，最多执行一次。
- 客户端崩溃会产生孤儿计算，可用消灭、再生、温和再生、到期处理。
- 基于消息通信：显式发送/接收消息，可支持队列和发布订阅。
- 消息代理：保存、路由、过滤、转换消息，代价是 broker 可靠性和性能要处理。
- 流数据：关注连续速率、延迟、抖动和播放连续性。
- 流 QoS：带宽、延迟、抖动、丢包率、同步误差。
- 丢包可靠性：ACK、超时重传、序列号、滑动窗口、FEC、缓冲和应用层去重。
