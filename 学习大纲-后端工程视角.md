## 总体主线

分布式系统可以压缩成一个问题链：

```text
多台机器如何像一个系统一样协作？
-> 它们如何通信？                      RPC / 消息 / TCP / UDP
-> 它们如何判断事件顺序？              时钟 / 同步 / happens-before
-> 它们如何复制数据且不乱？            一致性 / 复制 / Quorum
-> 它们如何在故障下继续工作？          容错 / 事务 / 共识 / 容灾
-> 它们如何支撑真实业务和大规模系统？  流媒体 / 云计算 / 大数据 / 大模型训练
```

后端工程视角下，这门课不是孤立理论课，而是在解释日常系统问题的底层原因：

- RPC 超时、重试、幂等、去重，背后是远程调用故障语义。
- 缓存、读写分离、多副本数据库，背后是一致性模型。
- MQ、异步任务、邮件系统，背后是持久通信。
- 分布式锁、事务、主从复制，背后是同步、互斥、2PC 和共识。
- 服务宕机、机房故障、脑裂，背后是容错、复制状态机和容灾。

## 模块一：基础概念

### 核心问题

- 什么是分布式系统？

- 开放性和透明性分别指什么？
- 策略和机制如何区分？
- 网络操作系统和基于中间件的分布式系统有什么区别？

### 后端工程连接

- 微服务系统对用户表现为一个整体，这就是“单一系统映像”的工程版本。

- 服务迁移、扩容、故障转移不影响调用方，是位置透明、迁移透明、故障透明的工程目标。

### 高频输出

- 分布式系统定义。
- 使用分布式系统的原因：经济性、性能、固有分布、可靠性、可扩展性。
- 透明性类型：访问、位置、迁移、重定向、复制、并发、故障。
- 开放性：良好接口、可移植、互操作、屏蔽异构性。

## 模块二：通信与 RPC

### 学习目标

理解远程调用为什么不能简单等价于本地调用，并能解释 RPC 的故障语义。该模块属于“算法、协议与真实系统问题”，需要重点结合后端经验深入理解。

### 核心问题

- TCP 和 UDP 各自适合什么通信场景？
- 同步/异步通信与持久/瞬时通信怎么区分？
- RPC 可能发生哪些故障？
- at-least-once 和 at-most-once 的区别是什么？
- 如何处理重复请求、响应丢失、客户端崩溃和服务器崩溃？

### 后端工程连接

- RPC 超时后重试可能导致重复下单，因此需要请求 ID、幂等键和服务端去重。
- at-least-once 类似“至少投递一次”，常见于 MQ 消费语义。
- at-most-once 类似“去重后最多执行一次”，依赖请求编号、历史缓存和重复响应。
- 客户端崩溃后的 orphan computation 类似请求方已经放弃，但服务端还在执行任务。
- 邮件、MQ 是持久通信；普通 HTTP/RPC 多数是瞬时通信。
- 工作中需要额外关注：超时预算、重试风暴、幂等语义、请求取消、服务端是否支持去重、调用链路上的代理和网关是否会重试。

### 高频输出

- RPC 五类故障：
  - 客户端不能定位服务器。
  - 请求消息丢失。
  - 响应消息丢失。
  - 服务器接收请求后故障。
  - 客户端发送请求后故障。
- at-most-once 的常见实现：
  - 每个请求带唯一请求 ID。
  - 服务端记录已处理请求。
  - 重复请求不重新执行，返回缓存结果或丢弃。
  - 客户端超时重试，但服务端保证不会重复执行。

### 易错提醒

不要简单写“TCP 是同步通信”。更严谨的说法是：

```text
TCP 是可靠的、面向连接的字节流传输协议；同步/异步通常描述应用层调用方式。
在阻塞式 socket 或典型同步 RPC 中，调用方会等待响应，因此可表现为同步通信。
```

## 模块三：时间与同步

### 学习目标

掌握没有全局时钟时如何判断事件顺序，并稳定完成计算题。该模块属于“算法、协议与真实系统问题”，重点是理解因果关系，而不是只会套公式。

### 核心问题

- 为什么物理时钟在分布式系统中不可靠？
- Lamport 时钟能表达什么，不能表达什么？
- 向量时钟为什么能判断并发？
- 如何根据消息传播过程标时间戳？
- 如何设计时间同步协议？

### 后端工程连接

- 日志排序、链路追踪、事件溯源都会遇到“不同机器时间不一致”的问题。
- 仅靠时间戳排序请求可能误判真实因果顺序。
- 分布式系统中更重要的是“因果顺序”而不是墙上时间。
- 版本向量、冲突检测、多活复制都和向量时钟思想相关。
- 工作中需要额外关注：NTP 只能改善物理时间偏差，不能提供可靠因果顺序；跨服务日志排查时，trace/span 关系通常比机器时间更可信。

### 高频输出

- Lamport 规则：
  - 本地事件：时钟加 1。
  - 发送消息：时钟加 1，并携带时间戳。
  - 接收消息：本地时钟设为 `max(local, message) + 1`。
- 向量时钟规则：
  - 本地事件：本进程分量加 1。
  - 发送消息：本进程分量加 1，并携带整个向量。
  - 接收消息：逐位取 max，再将本进程分量加 1。

### 易错提醒

- Lamport 时钟满足：如果 `a -> b`，则 `C(a) < C(b)`。
- 反过来不成立：`C(a) < C(b)` 不一定表示 `a -> b`。
- Lamport 可以通过 `(timestamp, process_id)` 构造全序，但这个全序不等于真实因果全序。
- 向量时钟的核心价值不是“构造全序”，而是判断因果关系和并发关系。

## 模块四：复制与一致性

### 学习目标

理解多副本系统为什么会不一致，以及不同一致性模型牺牲了什么、保证了什么。该模块属于“算法、协议与真实系统问题”，需要重点连接缓存、数据库和多副本读写。

### 核心问题

- 数据为中心和客户为中心一致性有什么区别？
- 顺序一致性、FIFO 一致性、因果一致性怎么判断？
- 门户网站为什么适合客户为中心的一致性？
- Quorum 如何保证读写冲突和写写冲突可检测？

### 后端工程连接

- Redis 缓存、CDN、读写分离数据库都在使用“更弱但更快”的一致性。
- 用户读到旧数据是最终一致性的典型表现。
- 单用户读自己刚写的数据，需要 read-your-writes consistency。
- 多副本数据库使用 quorum，本质是让读集合和写集合有交集。
- 工作中需要额外关注：一致性模型不是越强越好，强一致通常意味着更高延迟、更低可用性或更复杂的协调。

### 高频输出

- Quorum 条件：
  - 读写不冲突：`R + W > N`。
  - 写写不冲突：`W > N / 2`。
- 门户网站：
  - 读多写少。
  - 对短时间旧数据容忍度较高。
  - 适合客户为中心一致性或最终一致性。

### 易错提醒

回答一致性题时不要只说“最终一致性即可”，要补一句为什么：

```text
门户网站主要是读多写少，副本用于提升读取性能。用户通常可以容忍短时间读到旧内容，
因此无需所有副本在每次写后立即强一致，只需保证没有新写入时副本最终收敛。
```

## 模块五：容错、事务与共识

### 学习目标

理解故障模型、事务提交和共识协议之间的关系。该模块属于“算法、协议与真实系统问题”，需要重点理解协议为什么这样设计，以及故障时保证什么、不保证什么。

### 核心问题

- `2K + 1` 和 `3K + 1` 分别适用于什么故障模型？
- 2PC 如何提交事务？为什么会阻塞？
- Paxos 解决什么问题？为什么可能活锁？
- Raft 如何用 leader、term 和 log 保证复制状态机一致？

### 后端工程连接

- 数据库分布式事务常见 2PC，但协调者故障会带来阻塞风险。
- Raft 是很多工程系统的核心，例如 etcd、Consul、TiKV 的一部分。
- 多数派机制是服务注册中心、配置中心、元数据系统避免脑裂的基础。
- 容灾和多机房部署本质上是把故障域扩大后继续保证服务可用或数据不丢。
- 工作中需要额外关注：多数派协议解决的是一致性和脑裂问题，但会牺牲少数派分区的可用性；2PC 解决原子提交，但不是共识协议，且存在阻塞风险。

### 高频输出

- 唯一正确结果 + 错误节点返回随机值：
  - 至少 `2K + 1` 个副本。
  - 正确副本至少 `K + 1` 个，多数投票能压过 `K` 个错误结果。
- 传感器平均值 / 拜占庭式协同：
  - 常见要求 `3K + 1` 个节点。
  - 需要通过多轮交换识别一致信息，保证正确节点之间达成一致。
- Raft 三部分：
  - leader 选举。
  - 日志复制。
  - 安全性。

### 易错提醒

`2K + 1` 和 `3K + 1` 不能脱离故障模型背诵。考试答题时要先说明：

```text
如果错误节点只是返回任意值，而正确节点的结果唯一且相同，可以用多数投票，至少需要 2K+1。
如果错误节点可能向不同节点发送不同信息，并要求所有正确节点对一组值达成一致，则属于更强的拜占庭故障模型，常见要求是 3K+1。
```

## 模块六：应用与讲座题

### 学习目标

掌握若干应用场景中的分布式系统思想，用于问答题补分和理解真实系统。

### 核心问题

- 流媒体如何提高播放流畅性？
- 云计算为什么依赖虚拟化？
- Hadoop、Spark、YARN 解决什么问题？
- 大模型训练为什么是分布式系统问题？
- 数据中心为什么需要容灾？

### 后端工程连接

- 流媒体关注延迟、抖动、丢包恢复，本质是 QoS。
- YARN/Mesos 关注资源调度，类似现代 Kubernetes 调度问题。
- 大模型训练中的 AllReduce、参数同步、流水并行都依赖高性能通信。
- 容灾方案对应后端常见的备份、主备、多活、异地多活和故障切换。

### 高频输出

- 流媒体 QoS：
  - 比特率。
  - 会话创建最大延迟。
  - 端到端最大延迟。
  - 最大延迟抖动。
  - 最大往返延迟。
- 大模型训练挑战：
  - 数据并行。
  - 模型并行。
  - 通信瓶颈。
  - 故障恢复。
  - 存储和调度压力。

## 后续学习节奏

每一讲按下面结构推进：

```text
1. 先讲问题背景：它为什么在分布式系统中必然出现。
2. 用后端工程经验类比：RPC、缓存、数据库、MQ、配置中心等。
3. 给严谨定义：修正仓库答案中不够准确的说法。
4. 拆往年题：说明题目实际在考什么。
5. 给卷面模板：考试时如何写得短、准、稳。
6. 做 1-2 道练习：确认能算、能判、能写。
```

对于概念问答题，讲解会压缩为“定义 + 关键词 + 答题模板 + 易错点”。对于算法、协议和工程高频问题，会展开“模型假设 + 机制流程 + 正确性直觉 + 工程落地 + trade-off + 答题模板”。

建议学习顺序：

```text
基础概念
-> 通信与 RPC
-> 时间与同步
-> 复制与一致性
-> 容错、事务与共识
-> 应用与讲座题
```

## 第一阶段检查点

完成前 3 个模块后，应能做到：

- 用 3-5 句话严谨定义分布式系统。
- 区分开放性、透明性、策略和机制。
- 解释一次 RPC 调用中可能发生的所有故障。
- 说明 at-least-once 和 at-most-once 的实现差异。
- 独立完成 Lamport 和向量时钟计算题。
- 解释为什么机器物理时间不能直接作为事件因果顺序。

## 第二阶段检查点

完成一致性和容错模块后，应能做到：

- 判断顺序一致性、FIFO 一致性和因果一致性。
- 给出门户网站副本一致性设计。
- 对任意 `N` 写出合法的 quorum `R/W` 范围。
- 区分 `2K + 1` 和 `3K + 1` 的适用条件。
- 描述 2PC 在正常、超时、协调者宕机场景下的行为。
- 用自己的话说明 Raft 为什么依赖多数派和任期。

## 最终目标

最后形成一套个人答题材料：

- 高频概念题模板。
- 高频计算题套路。
- 高频协议流程图。
- 易错点修正版。
- 往年题对应索引。

这套材料的目标不是覆盖教材全部细节，而是确保考试主干稳定得分，同时让这些概念能回到真实后端系统设计中使用。
