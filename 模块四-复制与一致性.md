# 模块四：复制与一致性

> 时间紧时只抓五件事：为什么复制会不一致、数据为中心 vs 客户为中心、严格/线性/顺序/FIFO/因果一致性、weak/release/entry consistency、Quorum 两个公式。最新 PDF 明确列出了 linearizability 和带同步操作的一致性模型。

## 1. 本模块主线

复制能提升性能、可用性和容错能力，但副本越多，越容易出现不一致：

- 写入可能只到达部分副本。
- 不同副本收到写操作的顺序可能不同。
- 读请求可能读到旧副本。
- 强一致需要协调，会增加延迟并降低可用性。

核心句：

```text
一致性模型不是越强越好。
强一致通常意味着更多同步和更高延迟；弱一致牺牲部分新鲜度，换取性能、可用性和可扩展性。
```

后端类比：

- Redis 缓存、CDN、读写分离数据库都可能读到旧数据。
- 单用户“读自己刚写的数据”对应 read-your-writes consistency。
- Quorum 的本质是让读集合和写集合有交集。

## 2. 为什么需要复制

标准答案：

```text
分布式系统使用复制主要是为了提高可靠性、性能和可扩展性。
可靠性方面，某个副本故障时其他副本仍可继续服务；
性能方面，可将请求分散到多个副本，降低访问延迟并提升吞吐量；
可扩展性方面，可通过增加副本支持更多读请求或跨地域访问。
但复制会带来副本一致性问题，需要一致性协议或一致性模型约束副本行为。
```

## 3. 数据为中心 vs 客户为中心一致性

| 类型 | 关注点 | 典型场景 |
| --- | --- | --- |
| 数据为中心一致性 | 所有进程看到的数据读写顺序是否满足全局约束 | 共享数据库、分布式存储 |
| 客户为中心一致性 | 单个客户在不同副本之间移动时，自己看到的数据是否合理 | 移动用户、Web/CDN、读多写少系统 |

标准答案：

```text
数据为中心的一致性从整个数据存储角度定义所有读写操作应满足的顺序约束，
关注多个进程对共享数据的观察是否一致，如顺序一致性、因果一致性等。
客户为中心的一致性关注单个客户在访问不同副本时看到的结果是否符合自身预期，
典型保证包括读己之写、单调读、单调写和写后读等。
```

门户网站题标准答案：

```text
门户网站通常读多写少，副本主要用于提升读取性能和降低访问延迟。
用户一般可以容忍短时间读到旧内容，因此不必要求每次写入后所有副本立即强一致。
采用客户为中心一致性或最终一致性更合适，只需保证没有新写入时各副本最终收敛，并尽量保证单个用户体验一致。
```

易错点：

- 不要只写“最终一致性即可”，要说明“读多写少、容忍旧数据、最终收敛”。
- 客户为中心不是只服务一个用户，而是从单个客户观察到的行为定义一致性。

## 4. 数据为中心一致性模型

### 4.1 严格一致性

```text
任何读操作都返回最近一次写操作的结果。
```

现实问题：

- 依赖全局真实时间。
- 在分布式系统中通常难以实现。

### 4.2 线性一致性 Linearizability

一句话：

> 所有进程看到相同操作顺序，并且该顺序尊重真实时间上的先后关系。

标准答案：

```text
线性一致性要求所有进程看到所有共享访问按同一个顺序发生，并且如果一个操作在真实时间上先于另一个非重叠操作完成，
那么全局顺序中也必须保持这个先后关系。
它比顺序一致性更强，因为顺序一致性不要求遵守真实时间顺序。
```

易错点：

- 线性一致性不是严格一致性。严格一致性要求读到绝对最新值，依赖理想全局时钟；线性一致性用操作生效点解释并发对象行为。
- 考试中可按 PDF 表述记：所有进程看到相同顺序，并按非唯一全局时间戳排序。

### 4.3 顺序一致性

一句话：

> 所有进程看到同一个全局操作顺序，且每个进程自己的程序顺序被保留。

标准答案：

```text
顺序一致性要求所有进程看到的读写操作结果，就像这些操作按某个全局顺序依次执行一样。
这个全局顺序不一定等于真实时间顺序，但必须保持每个进程内部操作的程序顺序。
```

判断要点：

- 能否找到一个所有进程都接受的全局顺序。
- 该顺序必须保留每个进程自己的操作先后。

### 4.4 FIFO 一致性

一句话：

> 同一进程发出的写操作，所有进程必须按相同顺序看到；不同进程的写操作可被不同进程以不同顺序看到。

标准答案：

```text
FIFO 一致性要求同一个进程发出的写操作必须按照发出顺序被所有进程观察到。
但不同进程发出的写操作之间不要求所有进程观察到相同顺序。
因此 FIFO 一致性弱于顺序一致性。
```

判断要点：

- 只检查“同一个写进程”的写顺序是否被所有人保持。
- 不同写进程之间顺序不一致通常允许。

### 4.5 因果一致性

一句话：

> 有因果关系的写操作，所有进程必须按因果顺序看到；并发写操作可被不同进程以不同顺序看到。

标准答案：

```text
因果一致性要求所有进程以相同顺序观察到具有因果关系的写操作。
如果两个写操作是并发的、互相没有因果依赖，则不同进程可以以不同顺序观察到它们。
因果一致性强于 FIFO 一致性，但弱于顺序一致性。
```

判断要点：

- 如果写 B 读取了写 A 的结果，或由收到 A 后触发，则 A 因果先于 B。
- 有因果关系的写不能被反着看见。
- 并发写可以不同顺序。

## 5. 一致性强弱关系

常用关系：

```text
严格一致性 > 线性一致性 > 顺序一致性 > 因果一致性 > FIFO 一致性
```

注意：

- 这是常见考法里的粗略强弱关系。
- 不同教材对模型集合可能更细，但考试一般抓这条线即可。

## 6. 带同步操作的一致性模型

最新 PDF 还列出了 weak / release / entry consistency。这类模型的核心是：普通读写不一定随时一致，只在同步点附近保证一致。

| 模型 | 含义 |
| --- | --- |
| Weak consistency | 只有执行同步操作后，才能保证共享数据一致 |
| Release consistency | 退出临界区或释放同步变量时，把共享数据变为一致 |
| Entry consistency | 进入临界区或获取同步变量时，只同步该临界区相关数据 |

标准答案：

```text
弱一致性把一致性保证推迟到同步操作之后，普通读写之间不保证立即一致。
释放一致性要求进程退出临界区或释放同步变量时，将修改传播出去，使共享数据在释放点后可见。
入口一致性要求进程进入临界区或获取同步变量时，只同步与该同步变量相关的数据。
这类模型通过减少每次普通读写的同步开销，换取更好的性能。
```

易错点：

- weak/release/entry consistency 不是客户为中心一致性，它们属于“带同步操作”的一致性模型。
- release 在退出/释放时传播更新，entry 在进入/获取时拉取相关数据。

## 7. 客户为中心一致性模型

四个常见保证：

| 模型 | 含义 |
| --- | --- |
| 单调读 | 一旦读到某个版本，以后不会读到更旧版本 |
| 单调写 | 同一客户的写按发出顺序执行 |
| 读己之写 | 客户写入后，之后自己能读到该写入 |
| 写后读 | 客户基于某次读之后的写，应在该读对应版本之后执行 |

标准答案：

```text
客户为中心一致性关注单个客户在多个副本之间访问时的观察结果。
单调读保证客户读到某版本后不会再读到更旧版本；
单调写保证同一客户的写按顺序执行；
读己之写保证客户写入后自己后续能读到该结果；
写后读保证客户基于某次读发出的写，会在该读所依赖版本之后执行。
```

后端例子：

- 用户修改头像后自己刷新页面应看到新头像：读己之写。
- 用户连续修改资料，后一次修改不能被先一次覆盖：单调写。
- 用户翻页读消息，不应突然看到更旧状态：单调读。

## 8. Quorum 协议

变量：

- `N`：副本总数。
- `R`：一次读需要读到的副本数。
- `W`：一次写需要写入的副本数。

两个公式必须会：

```text
R + W > N      保证读集合和写集合有交集，避免读写冲突不可见。
W > N / 2      保证任意两个写集合有交集，避免写写冲突不可见。
```

标准答案：

```text
Quorum 协议通过要求读写操作访问一定数量的副本来保证一致性。
若 R + W > N，则任意一次读集合与最近一次写集合至少有一个共同副本，因此读操作有机会看到最新写入。
若 W > N/2，则任意两次写操作的写集合必然相交，因此可以检测或排序写写冲突。
```

例子：

| N | 可选 R/W | 是否满足 |
| --- | --- | --- |
| 3 | R=2, W=2 | `2+2>3` 且 `2>1.5` |
| 5 | R=2, W=3 | `2+3` 不大于 5，不满足读写交集 |
| 5 | R=3, W=3 | `3+3>5` 且 `3>2.5` |

易错点：

- `R + W > N` 必须是严格大于，不是大于等于。
- `W > N/2` 是为了写写集合相交。
- Quorum 不自动等于强一致，还需要版本号、冲突解决、读修复等机制配合。

## 8.5 复制协议实例

PPT 中不仅考一致性模型，也列了若干复制协议。答题时不用展开成工程实现，重点说清“谁接收写、如何传播、保证什么”。

| 协议 | 核心思想 | 优点 | 代价/风险 |
| --- | --- | --- | --- |
| Primary-backup | 主副本接收写，再同步或异步复制到备份 | 控制简单，容易排序写操作 | 主副本故障需切换，异步复制可能丢最新写 |
| Active replication | 所有副本按相同顺序执行同一请求 | 副本行为一致，容错强 | 需要全序组播或确定性执行 |
| Chain replication | 副本排成链，写从头传到尾，读通常从尾读 | 写传播顺序清晰，读到尾部确认结果 | 链上节点故障要重构 |
| ROWA | Read-one/write-all，读一个副本，写所有副本 | 读快，语义简单 | 写代价高，某副本不可用会影响写 |
| Quorum | 读写只需访问足够数量副本 | 在可用性和一致性间折中 | 需要版本号、冲突解决和读修复 |

标准答案：

```text
复制协议用于在多个副本之间传播读写操作并维护所需一致性。
主备复制由 primary 排序写操作并复制给 backup；主动复制要求所有副本以相同顺序执行同一请求；
链式复制把副本排成链，写沿链传播，读可从链尾获得已确认结果；
ROWA 要求写所有副本、读任一副本；Quorum 则通过读写集合交集在一致性和可用性之间折中。
```

### 8.5.1 Primary-backup

```text
Primary-backup 复制中，一个副本被指定为 primary，所有写请求先到 primary。
primary 负责给写操作排序，然后把更新传播给 backup。读请求可以只由 primary 处理，也可以在满足一致性要求时由 backup 处理。
如果 primary 故障，系统需要从 backup 中选出新的 primary，并确保新 primary 拥有足够新的状态。
```

答题模板：

```text
主备复制的优点是写操作顺序由 primary 统一决定，实现简单，容易维护顺序一致的更新流。
缺点是 primary 是性能瓶颈和故障切换关键点；如果采用异步复制，primary 故障时可能丢失尚未复制到 backup 的最新更新。
```

### 8.5.2 Active replication

```text
主动复制中，每个请求被发送给所有副本，所有副本都执行同一操作。
为了让副本状态保持一致，副本必须以相同顺序执行请求，并且操作最好是确定性的。
因此主动复制通常依赖全序组播或共识协议来保证请求顺序一致。
```

答题模板：

```text
主动复制通过让所有副本同时执行同一请求来提高容错能力。
它的关键条件是所有副本看到相同请求顺序，并且执行过程确定，否则副本会分叉。
优点是某个副本故障时其他副本仍有完整状态；缺点是全序广播或共识开销较大。
```

### 8.5.3 Chain replication

```text
链式复制把副本排成一条逻辑链。
写请求从链头进入，沿链依次传播到链尾；链尾确认后，说明该写已被整条链接收。
读请求通常从链尾读取，因为链尾包含已经按顺序完成传播的更新。
```

答题模板：

```text
链式复制的优点是写传播顺序清晰，链尾可提供已经确认的最新结果；
缺点是链上任一节点故障都需要重构链，写延迟也会受到链长度影响。
```

### 8.5.4 ROWA 与 Quorum

```text
ROWA 即 read-one/write-all。读操作只读任一副本，因此读开销低；
写操作必须更新所有副本，因此写开销高，并且某些副本不可用时写可能无法完成。
Quorum 是 ROWA 的放松版本：读不必只读一个，写也不必写全部，而是选择 R 和 W 使读写集合、写写集合有交集。
```

答题模板：

```text
ROWA 适合读多写少且希望读很快的场景，但写操作可用性较差；
Quorum 通过 R+W>N 和 W>N/2 保证读写、写写集合相交，在一致性、读性能、写性能和可用性之间折中。
```

易错点：

- Primary-backup 的重点是由 primary 排序，不是“只有一个副本保存数据”。
- Active replication 的重点是所有副本都执行请求，因此必须保证全序和确定性。
- Chain replication 中读链尾是为了读到已经完成链式传播的状态。
- ROWA 的读快是以写所有副本为代价；Quorum 是对 ROWA 的折中。

## 8.6 一致性模型应用题解题套路

这类题通常给出几个进程的读写程序，要求判断在顺序一致性、FIFO 一致性或因果一致性下可能出现什么结果。不要凭直觉写“最近写入”，要把每个读结果转换成顺序约束。

### 8.6.1 通用步骤

1. 把每条语句标成读写事件，例如 `W1(x=1)`、`R1(y)`。
2. 写出每个进程自己的程序顺序，例如 `W1(x=1) < R1(y)`。
3. 把读到的值翻译成约束：
   - `R(y)=0` 表示这个读没有看到 `W(y=1)`，在顺序一致性里可理解为 `R(y)` 排在 `W(y=1)` 之前。
   - `R(y)=1` 表示这个读看到了 `W(y=1)`，可理解为 `W(y=1)` 排在 `R(y)` 之前。
4. 对顺序一致性，检查所有约束能否组成一个共同全局顺序。
5. 对 FIFO 一致性，只检查同一个写进程发出的写是否被所有进程按同序看到；不同进程的写不要求全局同序。

### 8.6.2 多个写时的判断技巧

多个写时不要只看“最后一次写是谁”，要分别判断：

```text
顺序一致性：能不能排出一个共同全局顺序，使每个读都读到该顺序中离它最近的前一次写。
因果一致性：先找写与写之间的因果关系；有因果关系的写不能被任何进程反着观察，并发写可以被不同进程以不同顺序观察。
FIFO 一致性：只看同一个写进程连续发出的写是否按原顺序被所有进程看到。
```

常见因果关系来源：

```text
1. 同一进程内，前面的写因果先于后面的写。
2. 进程先读到某个写，再执行自己的写，则被读到的写因果先于后面的写。
3. 消息发送和接收也会传递因果关系。
```

读值转约束时可这样写：

```text
R(x)=a 表示该读看到的是 W(x)=a。
在顺序一致性里，R(x)=a 要求 W(x)=a 排在该读之前；
并且其他覆盖 x 的写如果会使该读不再读到 a，就必须排在该读之后，或排在 W(x)=a 之前。
```

考试作答时可以先尝试构造全局顺序：

```text
能构造出一个保持各进程程序顺序、并解释所有读值的全局顺序，则满足顺序一致性。
构造不出时，最好写出形成矛盾的环，例如 A < B < C < A。
```

### 8.6.3 例题：多个写下的因果一致性和顺序一致性

题目中所有操作都作用在同一个变量 `x`，初值可看成 `x=0`。

#### 模型 A

```text
P1: W(x)=a

P2: R(x)=a, W(x)=b

P3: R(x)=b, R(x)=a

P4: R(x)=b, R(x)=a
```

因果一致性判断：

```text
P2 先读到 a，再写 b，所以有因果链：
W(x)=a -> R2(x)=a -> W(x)=b

因此 W(x)=a 因果先于 W(x)=b。
因果一致性要求所有进程都按这个因果顺序观察这两个写。
但 P3 和 P4 都先读到 b，再读到 a，等于反着观察了 W(a) 和 W(b)。
所以模型 A 不满足因果一致性。
```

顺序一致性判断：

```text
由 P2 的 R(x)=a 且随后 W(x)=b，可得到全局顺序中需要：
W(a) < R2(a) < W(b)
因此 W(a) < W(b)。

但 P3 先 R(x)=b 后 R(x)=a。
如果只有 W(a) 和 W(b) 两次写，要先读到 b 再读到 a，
就需要 W(b) 排在第一次读之前，而 W(a) 排在第二次读之前且覆盖 b，
等价于需要 W(b) < W(a)。

这与 W(a) < W(b) 矛盾。
所以模型 A 不满足顺序一致性。
```

结论：

```text
模型 A 不满足因果一致性，也不满足顺序一致性。
```

#### 模型 B

```text
P1: W(x)=a, W(x)=c

P2: R(x)=a, W(x)=b

P3: R(x)=c, R(x)=b

P4: R(x)=a, R(x)=b
```

因果一致性判断：

```text
P1 内部有 W(a) -> W(c)。
P2 读到 a 后写 b，因此有 W(a) -> W(b)。

所以 W(a) 是 W(c) 和 W(b) 的共同因果前驱。
但 W(c) 和 W(b) 之间没有必然因果关系，可以并发。

P3 看到 c 再看到 b，是在两个并发写之间选择一种观察顺序，不违反因果一致性。
P4 看到 a 再看到 b，也没有在看到 b 或 c 之后又读回 a。
所以模型 B 可以满足因果一致性。
```

顺序一致性判断：

可以构造一个共同全局顺序：

```text
W(a)
R2(x)=a
R4(x)=a
W(c)
R3(x)=c
W(b)
R3(x)=b
R4(x)=b
```

这个顺序保持了各进程程序顺序：

```text
P1: W(a) < W(c)
P2: R2(a) < W(b)
P3: R3(c) < R3(b)
P4: R4(a) < R4(b)
```

并且能解释所有读值：

```text
R2(a)、R4(a) 都发生在 W(c)、W(b) 之前，所以读到 a。
R3(c) 发生在 W(c) 之后、W(b) 之前，所以读到 c。
R3(b)、R4(b) 发生在 W(b) 之后，所以读到 b。
```

结论：

```text
模型 B 满足因果一致性，也满足顺序一致性。
```

易错点：

- 因果一致性不要求所有并发写有统一顺序；只要求因果相关的写不被反向观察。
- 顺序一致性不看真实时间位置，只看能否构造一个合法全局顺序。
- 读到旧值不一定违规；关键看这个旧读能否放进某个全局顺序里解释。

### 8.6.4 例题：两个进程互相 kill

题目：

```text
初始 x = 0, y = 0

P1: x = 1
    if (y == 0) kill(P2)

P2: y = 1
    if (x == 0) kill(P1)
```

记号：

```text
A = W1(x=1)
B = R1(y)
C = W2(y=1)
D = R2(x)

程序顺序：
A < B
C < D
```

#### 顺序一致性下

顺序一致性要求存在一个所有进程都认可的全局执行顺序，并且保留每个进程自己的程序顺序。

如果两个进程都 kill 对方，则：

```text
P1 kill P2 表示 R1(y)=0，所以 B < C。
P2 kill P1 表示 R2(x)=0，所以 D < A。
再加上程序顺序 A < B 和 C < D。

得到 A < B < C < D < A，形成环，矛盾。
```

因此：

```text
顺序一致性下，不可能两个进程互相 kill。
可能结果是：P1 kill P2、P2 kill P1、或者二者都不 kill。
如果题目默认两个进程都已经执行完各自写操作，再讨论读结果，则常见可写结果是二者都不 kill。
但最核心结论是：双杀不满足顺序一致性。
```

可给的全局顺序例子：

```text
都不 kill：
A = W1(x=1), C = W2(y=1), B = R1(y)=1, D = R2(x)=1

只有 P1 kill P2：
A = W1(x=1), B = R1(y)=0，此时 P1 kill P2

只有 P2 kill P1：
C = W2(y=1), D = R2(x)=0，此时 P2 kill P1
```

#### FIFO 一致性下

FIFO 一致性只要求同一个进程发出的写操作被所有进程按发出顺序看到。这个例子里每个进程只有一个写：

```text
P1 只写 x=1。
P2 只写 y=1。
```

所以 FIFO 对这两个写之间没有全局排序要求。可以出现：

```text
P1 自己已经执行 W1(x=1)，但还没看到 P2 的 W2(y=1)，于是 R1(y)=0，kill(P2)。
P2 自己已经执行 W2(y=1)，但还没看到 P1 的 W1(x=1)，于是 R2(x)=0，kill(P1)。
```

这个结果满足 FIFO，因为没有违反任何“同一写进程的写顺序”；但它不满足顺序一致性，因为不存在一个共同全局顺序能让两个读都读到 0。

标准答案模板：

```text
顺序一致性要求所有进程看到同一个全局顺序，并保持各进程程序顺序。
若两个进程都互相 kill，则 P1 读 y=0 要求 R1(y) 在 W2(y=1) 之前，
P2 读 x=0 要求 R2(x) 在 W1(x=1) 之前；
再结合 W1(x=1) 在 R1(y) 之前、W2(y=1) 在 R2(x) 之前，会形成循环顺序，因此双杀不可能。

FIFO 一致性只保证同一进程的写按顺序被看到，不要求不同进程的写有统一全局顺序。
由于 P1 和 P2 各自只有一个写，两个进程可以暂时都看不到对方的写，从而都读到 0 并互相 kill。
所以双杀可以满足 FIFO，但不满足顺序一致性。
```

易错点：

- FIFO 不等于“读到最近写入”。它只约束同一写进程的写传播顺序。
- 顺序一致性不要求真实时间顺序，但要求存在一个共同全局顺序。
- FIFO 题不要强行写一个全局 interleaving；应写不同进程可以看到不同写顺序。

## 9. 常见答题模板

### 9.1 为什么复制会导致一致性问题

```text
复制会把同一数据存放在多个副本上，提高性能和可靠性。
但写操作传播到不同副本需要时间，不同副本可能收到不同顺序的更新，读请求也可能访问到旧副本。
因此多副本系统需要一致性模型说明允许哪些旧读、乱序或冲突，并通过协议协调副本状态。
```

### 9.2 门户网站副本一致性设计

```text
门户网站以读请求为主，副本主要用于提升读取性能和降低访问延迟。
这类系统通常能容忍短时间旧数据，因此不需要每次写入都同步到所有副本后再返回。
可采用客户为中心一致性或最终一致性，保证副本最终收敛，并尽量为单个用户提供单调读和读己之写等体验。
```

## 10. 考前最小背诵清单

- 复制优点：可靠性、性能、可扩展性。
- 复制缺点：复制透明性成本、一致性问题、更新代价高、处理不好会影响可用性。
- 复制问题：写传播延迟、读到旧副本、不同副本写顺序不同。
- 数据为中心：从共享数据角度约束所有读写。
- 客户为中心：从单个客户观察角度约束读写。
- 线性一致性：所有进程看到同一顺序，并尊重真实时间先后。
- 顺序一致性：所有进程看到同一个全局顺序，保留各进程程序顺序。
- FIFO：同一进程写顺序所有人一致，不同进程写顺序可不同。
- 因果一致性：有因果关系的写顺序一致，并发写可不同。
- weak/release/entry：一致性与同步操作绑定。
- Quorum：`R + W > N`，`W > N/2`。
- 复制协议：primary-backup、active replication、chain replication、ROWA、Quorum。
