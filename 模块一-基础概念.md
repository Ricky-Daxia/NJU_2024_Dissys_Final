# 模块一：基础概念

> 时间紧时只背这一版：定义、目标、原因、开放性、透明性、系统类型、架构组织、进程线程和代码迁移。最新 PDF 明确把“分布式操作系统/网络操作系统/中间件系统”和“架构组织形式”列入复习范围。

## 1. 分布式系统定义

标准答案：

```text
分布式系统是若干独立计算机的集合，这些计算机通过网络通信和协作。
从用户或应用程序角度看，它们像一个统一的系统一样提供服务。
它的核心目标是资源共享、提高性能和可靠性，并尽量屏蔽底层机器、网络和操作系统的异构性。
```

关键词：

- 独立计算机。
- 网络通信。
- 协同工作。
- 对用户呈现单一系统映像。

易错点：

- 不能只写“多台机器连在一起”，必须写“对用户像一个系统”。
- 分布式系统不等于普通网络系统，重点是统一协作和透明性。

## 2. 为什么使用分布式系统

必背五点：

| 原因 | 含义 |
| --- | --- |
| 经济性 | 多台普通机器比一台大型机更便宜 |
| 性能 | 多节点并行处理，提高吞吐量和计算能力 |
| 固有分布性 | 用户、数据或设备本来就在不同地点 |
| 可靠性 | 部分节点故障时，系统仍可继续服务 |
| 可扩展性 | 可逐步增加节点、服务或副本来扩容 |

标准答案：

```text
使用分布式系统主要因为经济性、性能、固有分布性、可靠性和可扩展性。
经济性指用多台廉价机器替代昂贵大型机；性能指任务可并行执行；
固有分布性指应用本身跨地域或跨组织；可靠性指单点故障不一定导致整体失败；
可扩展性指系统可通过增加节点逐步增长。
```

## 3. 分布式系统目标

最新 PDF 的四个目标：

| 目标 | 含义 |
| --- | --- |
| Making resources available | 让远程资源、数据和服务可被共享访问 |
| Distribution transparency | 隐藏资源分布、复制、迁移、故障等细节 |
| Openness | 通过标准接口和协议支持互操作与扩展 |
| Scalability | 在用户数、节点数、地域范围扩大时仍能工作 |

标准答案：

```text
分布式系统的目标包括资源共享、分布透明性、开放性和可扩展性。
资源共享使用户能够访问远程数据、设备和服务；
透明性隐藏资源位置、访问方式、复制、迁移和故障等分布式细节；
开放性依赖标准接口和协议支持互操作与扩展；
可扩展性要求系统在规模、地域和管理复杂度增加时仍能有效运行。
```

## 4. 开放性

一句话：

> 开放性指系统通过清晰、标准、良好定义的接口，使不同组件和系统容易互操作、移植、扩展和替换。

必背点：

- 良好定义的接口。
- 互操作性。
- 可移植性。
- 可扩展性。
- 屏蔽底层异构性。

标准答案：

```text
分布式系统的开放性主要体现在接口和协议规范化。
开放系统中的组件应能被其他系统使用或集成，不依赖特定硬件、操作系统或网络环境。
开放性带来互操作性、可移植性和可扩展性，使系统更容易替换组件、接入新服务和支持异构环境。
```

后端类比：

- REST/gRPC/IDL 是接口开放性的体现。
- 服务注册发现、版本兼容、API 网关帮助不同服务协作。

易错点：

- 开放性不是“所有资源都公开访问”，而是“接口规范、可互操作、可扩展”。

## 5. 透明性

一句话：

> 透明性指系统隐藏分布式实现细节，让用户像使用单机系统一样使用分布式资源。

常考类型：

| 类型 | 含义 |
| --- | --- |
| 访问透明性 | 隐藏本地和远程资源访问方式差异 |
| 位置透明性 | 隐藏资源物理位置 |
| 迁移透明性 | 资源迁移不影响访问方式 |
| 重定位透明性 | 资源使用过程中移动也不影响访问 |
| 复制透明性 | 隐藏多个副本的存在 |
| 并发透明性 | 并发访问不破坏资源一致状态 |
| 故障透明性 | 尽量隐藏或恢复部分故障 |

标准答案：

```text
透明性是指分布式系统隐藏资源分布和实现细节，使用户感觉自己在使用一个统一系统。
常见透明性包括访问透明性、位置透明性、迁移透明性、重定位透明性、复制透明性、并发透明性和故障透明性。
例如位置透明性使用户无需知道资源在哪台机器上，复制透明性使用户无需知道系统有多少副本。
```

后端类比：

- 服务扩容后调用方不感知具体实例：位置透明性。
- CDN/缓存/数据库副本对用户隐藏：复制透明性。
- 主从切换后业务尽量无感：故障透明性。

易错点：

- 故障透明性不是系统永远不失败，而是尽量隐藏、恢复或降低故障影响。
- 透明性越强不一定越好，可能增加实现复杂度和性能成本。

## 6. 策略与机制

一句话：

> 策略决定“做什么”，机制提供“怎么做”。

对比：

| 项目 | 策略 | 机制 |
| --- | --- | --- |
| 关注点 | 目标和选择规则 | 实现手段 |
| 回答问题 | 选择什么方案 | 如何支持该方案 |
| 变化频率 | 更容易变化 | 相对稳定 |

标准答案：

```text
机制和策略分离是开放分布式系统获得灵活性的关键。
策略描述系统应该做什么，例如选择延迟最低的副本处理请求。
机制描述系统如何做，例如提供服务发现、健康检查、负载均衡和请求路由能力。
二者分离后，可以在不修改底层机制的情况下调整策略。
```

例子：

- 策略：优先访问同机房副本；机制：服务发现 + 多副本路由。
- 策略：超过 200ms 切换备用节点；机制：超时检测 + 故障转移。
- 策略：VIP 请求优先处理；机制：优先级队列 + 资源隔离。

## 7. 分布式操作系统、网络操作系统和中间件系统

最新 PDF 明确要求区分三类系统。

| 类型 | 特点 |
| --- | --- |
| 分布式操作系统 | 多台机器由一个统一 OS 管理，尽量让多机像单机一样工作 |
| 网络操作系统 | 多台机器联网，但用户通常仍能感知不同机器 |
| 基于中间件的分布式系统 | 通过中间件屏蔽异构性和位置差异，对上提供统一系统视图 |

标准答案：

```text
分布式操作系统试图在操作系统层统一管理多台机器，使用户尽量感觉不到多机存在。
网络操作系统通过网络连接多台计算机，提供远程登录、文件共享等功能，但用户往往仍需知道资源位于哪台机器。
基于中间件的分布式系统在操作系统和应用之间加入中间层，通过统一接口、命名、通信和服务发现屏蔽异构性，
使多个独立节点对应用表现为一个统一系统。
```

易错点：

- 网络操作系统不一定提供单一系统映像，用户常常能感知远程机器。
- 中间件系统不是替代操作系统，而是在 OS 和应用之间提供分布式抽象。

## 8. 分布式系统架构与组织形式

### 8.1 架构风格

标准答案：

```text
分布式系统架构风格描述系统组件如何划分以及组件之间如何交互。
常见风格包括分层架构、对象/服务架构、以数据为中心的架构和事件驱动架构。
考试中通常不需要展开到具体框架，重点说明架构风格用于组织组件、通信关系和职责边界。
```

PPT 中常见四类说法：

| 架构风格 | 核心含义 | 后端类比 |
| --- | --- | --- |
| Layered | 系统按层组织，上层调用下层 | OSI、三层 Web 架构 |
| Object-based | 组件通过远程对象或服务接口交互 | RPC、微服务 |
| Publish/subscribe | 发布者和订阅者通过事件解耦 | MQ topic、事件总线 |
| Shared dataspace | 进程通过共享数据空间间接通信 | Tuple space、共享黑板 |

答题角度：

```text
如果题目问“分布式系统有哪些架构风格”，不要只列名词。
应先说架构风格描述组件如何组织和交互，再分别解释典型交互方式、优点和代价。
```

四类风格展开：

| 风格 | 如何工作 | 优点 | 代价/易错点 |
| --- | --- | --- | --- |
| Layered | 系统被拆成若干层，上层只通过接口使用下层服务 | 职责清晰，便于替换和维护 | 严格分层可能增加调用开销，跨层优化困难 |
| Object-based | 每个对象或服务封装状态和方法，其他组件通过远程调用访问它 | 封装好，接口清楚，适合 RPC/RMI | 远程对象引用、对象生命周期和调用故障要处理 |
| Publish/subscribe | 发布者把事件发到主题或事件通道，订阅者按兴趣接收 | 时间、空间和同步解耦，适合事件驱动系统 | 事件顺序、重复投递、丢失和调试链路更复杂 |
| Shared dataspace | 进程不直接互调，而是读写共享数据空间完成协作 | 解耦强，适合黑板系统和协作式计算 | 共享数据空间本身可能成为瓶颈，一致性要处理 |

标准答案模板：

```text
分布式系统的架构风格用于描述系统组件如何划分、组件之间通过什么方式交互以及职责边界如何确定。
常见风格包括分层架构、对象/服务式架构、发布订阅架构和共享数据空间架构。
分层架构按层次调用，优点是职责清晰；对象式架构通过远程对象或服务接口交互，便于封装；
发布订阅架构通过事件通道解耦发送方和接收方；共享数据空间架构通过共同读写数据空间间接通信。
不同风格关注的权衡不同：分层强调可维护性，对象式强调接口封装，发布订阅强调解耦，共享数据空间强调间接协作。
```

### 8.1.1 中间件 interceptor 与自管理系统

```text
Interceptor 是插入在请求调用路径中的中间件机制，用于在不改变业务组件主体逻辑的情况下增加通用功能。
例如安全认证、压缩、日志、监控、协议转换和重试都可以通过 interceptor 完成。
自管理系统则通过反馈控制循环运行：监控系统状态，分析是否偏离目标，规划调整策略，再执行配置或资源变更。
```

知识点展开：

```text
中间件的价值是为上层应用提供统一的分布式抽象，例如远程调用、命名、事务、安全、复制和服务发现。
Interceptor 是中间件实现可扩展性的常见机制。调用请求经过客户端 stub、网络、服务端 stub 或代理时，
中间件可以在这些路径上插入额外处理逻辑。这样应用代码不必关心日志、认证、压缩、监控等横切功能。
```

例子：

| 场景 | interceptor 做什么 |
| --- | --- |
| 安全认证 | 在请求发出前附加 token，在服务端执行业务前校验身份 |
| 日志和追踪 | 自动记录 trace id、耗时、请求来源和错误信息 |
| 协议转换 | 在客户端和服务端使用不同协议或数据格式时进行转换 |
| 重试/超时 | 在调用失败时按策略进行重试或快速失败 |

自管理系统答题模板：

```text
自管理系统是指系统能够根据运行状态自动调整自身行为的分布式系统。
它通常采用反馈控制模型：先监控系统状态，如负载、延迟和故障；再分析当前状态是否偏离目标；
然后规划调整动作，如扩容、迁移、限流或切换副本；最后执行这些动作并继续监控结果。
这种设计可以提升系统适应负载变化和故障变化的能力，但也需要避免错误监控、错误决策和频繁震荡。
```

易错点：

- interceptor 是机制，不是具体策略；策略决定何时限流或切换，interceptor 提供插入逻辑的位置。
- 自管理系统不是“完全不用人工”，而是把监控、分析、计划、执行部分自动化。
- 自管理系统的核心不是“用了 AI”，而是“有反馈控制回路”。

### 8.2 系统组织形式

最新 PDF 给出三类组织：

| 组织形式 | 含义 | 优缺点 |
| --- | --- | --- |
| Centralized | 有中心服务器或中心协调者 | 简单、易管理，但可能有瓶颈和单点故障 |
| Decentralized | 节点对等协作，无固定中心 | 可扩展、鲁棒，但查找和协调更复杂 |
| Hybrid | 中心化与去中心化结合 | 折中性能、管理和容错 |

标准答案：

```text
分布式系统组织形式可分为 centralized、decentralized 和 hybrid。
集中式组织依赖中心节点，结构简单但可能形成瓶颈和单点故障；
去中心化组织中节点对等协作，扩展性和鲁棒性更好，但查找、路由和一致性维护更复杂；
混合式组织结合二者，例如用中心节点做索引或协调，用对等节点承担数据传输。
```

### 8.3 Client-server 与 P2P

```text
Client-server 模式中客户端请求服务，服务器集中处理和管理资源，优点是控制简单，缺点是服务器可能成为瓶颈。
P2P 模式中节点既可请求资源也可提供资源，更去中心化，扩展性较好，但节点动态变化会增加查找和管理复杂度。
```

PPT 对 P2P 的补充分法：

| 类型 | 含义 |
| --- | --- |
| Structured P2P | 使用 DHT、逻辑环等结构定位资源 |
| Unstructured P2P | 通过泛洪、随机游走或局部邻居关系查找 |
| Super-peer | 部分能力强的节点承担索引、路由或协调职责 |
| Hybrid | 中心节点负责发现或索引，节点之间直接传输数据 |

BitTorrent 就适合答成混合式：tracker 或 DHT 帮助发现 peer，数据块在 peer 之间直接交换。

答题角度：

```text
P2P 的核心不是“没有服务器”，而是节点同时扮演客户端和服务端。
结构化 P2P 重点在可预测查找，非结构化 P2P 重点在灵活和简单，super-peer 与 hybrid 则是在完全去中心和集中式之间折中。
```

四类 P2P 展开：

| 类型 | 查找方式 | 适合场景 | 主要问题 |
| --- | --- | --- | --- |
| Structured | 按 DHT、环或树等覆盖结构定位 key | 大规模、需要高效查找资源 | 节点频繁加入离开时维护结构成本高 |
| Unstructured | 泛洪、随机游走或邻居转发 | 节点动态强、资源查找要求不严格 | 查找效率低，可能产生大量消息 |
| Super-peer | 普通节点连到能力更强的 super peer，由其维护索引或路由 | 节点能力差异明显的系统 | super peer 可能成为局部瓶颈或故障点 |
| Hybrid | 中心节点做发现/索引，数据由 peer 直接传输 | 内容分发、文件共享 | 中心节点仍需容错，且要处理 peer 动态变化 |

标准答案模板：

```text
P2P 系统中节点既是服务消费者也是服务提供者。
结构化 P2P 使用 DHT 等确定结构组织节点，可按 key 高效查找资源，但维护复杂；
非结构化 P2P 通过泛洪或随机搜索查找资源，实现简单但效率低；
super-peer 让能力强的节点承担索引或路由职责，是去中心化和集中式之间的折中；
混合式 P2P 通常由中心节点完成发现或索引，而数据在 peer 之间直接传输，例如 BitTorrent。
```

## 9. 三层 Client-Server 架构

三层：

| 层次 | 职责 |
| --- | --- |
| User-interface Layer | 用户界面和输入输出 |
| Processing Layer | 应用逻辑和业务规则 |
| Data Layer | 数据存储和管理 |

标准答案：

```text
三层 client-server 架构包括用户界面层、处理层和数据层。
用户界面层负责与用户交互，处理层负责应用逻辑和业务规则，数据层负责存储和管理应用数据。
这种划分有利于职责分离、独立扩展和维护。
```

后端类比：

- 前端页面/App：用户界面层。
- 后端服务：处理层。
- MySQL/Redis/对象存储：数据层。

## 10. 其他概念速记

### 10.1 进程与线程

```text
进程和线程都是程序执行单元，都可并发执行。
进程拥有独立地址空间和资源，隔离性强但切换成本较高；
线程共享同一进程资源，切换成本较低但隔离性较弱。
Web 应用采用多线程可同时处理多个请求，并在部分线程等待 I/O 时继续执行其他任务。
```

最新 PDF 里的关键表达：

```text
线程是 lightweight process，是共享同一地址空间的一条执行流。
两个进程访问同一个虚拟地址，看到的可能是不同内容；同一进程内两个线程访问同一地址，看到的是同一份内容。
```

线程实现方式：

| 类型 | 优点 | 缺点 |
| --- | --- | --- |
| User-level thread | 创建和切换快，不必频繁进入内核 | 一个线程阻塞可能影响整个进程 |
| Kernel-level thread | 内核可感知并调度每个线程 | 创建、切换和同步成本更高 |
| Mixed approach | 兼顾用户态轻量和内核调度能力 | 实现更复杂 |

知识点展开：

```text
User-level thread 的线程管理在用户态完成，创建和切换不需要陷入内核，因此速度快。
但如果内核只看到一个进程，那么一个用户级线程执行阻塞系统调用时，可能导致整个进程被阻塞。
Kernel-level thread 由内核直接管理，内核能把同一进程中的不同线程调度到不同 CPU 上，也能在某个线程阻塞时继续运行其他线程。
代价是线程操作需要内核参与，开销较大。
Mixed approach 试图把多个用户级线程映射到若干内核线程上，兼顾效率和并行能力。
```

答题模板：

```text
用户级线程由用户态线程库管理，优点是创建和切换快，缺点是内核不可见，阻塞调用可能影响整个进程。
内核级线程由操作系统内核管理，优点是可被内核独立调度并利用多处理器，某个线程阻塞不一定阻塞整个进程；
缺点是创建、切换和同步开销较高。
混合线程模型把用户级线程映射到内核线程上，试图在轻量切换和内核调度能力之间折中，但实现更复杂。
```

服务端状态：

| 类型 | 含义 |
| --- | --- |
| Stateless server | 不保存客户端会话状态，每个请求自包含 |
| Stateful server | 保存客户端状态，便于连续交互但恢复更复杂 |
| Soft-state server | 状态带租约或超时，客户端需刷新，故障后可逐步重建 |

服务端状态展开：

```text
Stateless server 不保存客户端会话状态，每个请求必须携带完成处理所需的全部信息。
它易于扩展、负载均衡和故障恢复，因为任意服务器实例都可处理请求。
缺点是请求可能更大，重复携带上下文，某些连续交互实现不方便。
Stateful server 会保存客户端状态，例如会话、锁、打开文件位置或事务上下文。
它能简化连续操作，但服务器故障后需要恢复状态，否则客户端语义会被破坏。
Soft-state server 介于两者之间，服务器保存状态但状态有超时或租约，客户端需要定期刷新。
如果服务器故障或客户端消失，状态最终会过期，恢复难度低于强 stateful。
```

答题模板：

```text
无状态服务器不保存客户端会话状态，每个请求自包含，因此容易扩展、负载均衡和故障恢复；
有状态服务器保存客户端相关状态，能支持连续会话、锁和事务等语义，但故障恢复更复杂；
软状态服务器保存带超时或租约的状态，客户端需定期刷新，状态丢失后可重建，是无状态和有状态之间的折中。
```

### 10.2 结构化 P2P 与非结构化 P2P

```text
结构化 P2P 按特定分布式数据结构组织节点，如 DHT 或逻辑环，可根据 key 高效定位资源。
优点是查找效率高、可扩展性好，缺点是维护复杂。
非结构化 P2P 中节点随机或局部建立邻居关系，实现简单、灵活，但查找通常依赖泛洪或随机搜索，效率较低。
```

### 10.3 强迁移与弱迁移

```text
强迁移会迁移代码、数据和执行状态，使程序可从原执行点继续运行。
弱迁移只迁移代码和必要数据，迁移后通常从预定义入口重新开始。
强迁移透明性更好但实现复杂，弱迁移实现较简单但对程序结构有更多要求。
```

### 10.3.1 代码迁移 vs 虚拟机迁移

```text
代码迁移是把程序代码以及必要数据移动到另一台机器上执行。
虚拟机迁移是把整个虚拟机运行环境迁移到另一台物理机上，通常包括虚拟 CPU、内存、磁盘和网络状态。
二者都能把计算移动到别处，但粒度不同：代码迁移粒度小，虚拟机迁移粒度大。
```

| 项目 | 代码迁移 | 虚拟机迁移 |
| --- | --- | --- |
| 迁移对象 | 程序代码、数据，强迁移还包括执行状态 | 完整虚拟机及其运行状态 |
| 优点 | 粒度小、灵活、传输量较小 | 对应用透明，环境一致性好 |
| 缺点 | 依赖目标机器运行环境，兼容性和安全性要处理 | 迁移数据量大，性能开销和管理复杂度高 |
| 适合场景 | 移动代理、把计算移动到数据附近 | 云平台负载均衡、硬件维护、故障迁移 |

答题模板：

```text
代码迁移的优点是迁移粒度较小、灵活，适合把计算逻辑移动到数据或资源附近执行；
缺点是目标节点必须具备兼容的运行环境，并且要处理安全、权限和依赖问题。
虚拟机迁移迁移的是完整运行环境，对应用更透明，不需要应用感知底层物理机变化；
缺点是迁移状态量大，网络、存储和内存状态处理复杂，且虚拟化本身有一定性能开销。
```

### 10.4 虚拟化

```text
虚拟化是在底层硬件和上层程序之间加入抽象层，让程序看到一个逻辑运行环境。
Process virtual machine 只为单个进程提供运行环境，例如语言运行时或解释器；
virtual machine monitor 则在硬件之上模拟完整机器，使多个操作系统实例能共享同一物理机器。
云计算依赖虚拟化实现资源池化、隔离、弹性伸缩和快速迁移。
```

知识点展开：

| 类型 | 抽象对象 | 例子 | 重点 |
| --- | --- | --- | --- |
| Process VM | 单个进程的运行环境 | JVM、语言运行时、解释器 | 屏蔽 OS/硬件差异，让程序可移植 |
| Virtual Machine Monitor | 完整机器或硬件平台 | Hypervisor、虚拟机 | 多个 OS 共享硬件，提供隔离和资源管理 |

答题模板：

```text
虚拟化通过在真实硬件和上层软件之间加入抽象层，使上层看到的是虚拟资源而不是真实物理资源。
进程虚拟机面向单个进程，提供统一运行环境，例如语言运行时；
虚拟机监控器面向完整操作系统，模拟 CPU、内存、磁盘和网络等硬件资源，使多个虚拟机能运行在同一物理机器上。
云计算依赖虚拟化实现资源池化、隔离、弹性分配、迁移和按需使用。
```

易错点：

- 虚拟化不是简单“远程登录机器”，而是资源抽象和隔离机制。
- 进程虚拟机不等于完整虚拟机，前者服务单个进程，后者可运行完整操作系统。

虚拟化解决的问题：

```text
虚拟化主要解决物理资源利用率低、应用隔离困难、部署迁移复杂和弹性扩展困难等问题。
通过把物理 CPU、内存、存储和网络抽象成虚拟资源，一台物理机可以运行多个隔离的虚拟环境；
系统可以按需创建、销毁、迁移和扩展虚拟机或容器。
云计算建立在这些能力之上，通过资源池化和自动化管理向用户提供 IaaS、PaaS 或 SaaS 等服务。
```

## 11. 考前最小背诵清单

- 定义：独立计算机集合 + 网络通信 + 对用户像单一系统。
- 目标：资源共享、透明性、开放性、可扩展性。
- 使用原因：经济性、性能、固有分布性、可靠性、可扩展性。
- 开放性：接口规范、互操作、可移植、可扩展、屏蔽异构。
- 透明性：访问、位置、迁移、重定位、复制、并发、故障。
- 策略/机制：策略是做什么，机制是怎么做。
- 三类系统：分布式操作系统、网络操作系统、基于中间件的系统。
- 组织形式：centralized、decentralized、hybrid。
- 架构风格：layered、object-based、publish/subscribe、shared dataspace。
- 三层架构：用户界面层、处理层、数据层。
- P2P：structured、unstructured、super-peer、hybrid。
- 网络操作系统 vs 中间件：前者用户更能感知机器，后者提供统一系统视图。
- 线程实现：user-level、kernel-level、mixed；服务端状态：stateless、stateful、soft-state。
- 代码迁移 vs 虚拟机迁移：前者迁移程序，粒度小；后者迁移完整运行环境，对应用更透明。
- 虚拟化与云计算：虚拟化提供资源抽象、隔离、迁移和弹性，是云计算基础。
